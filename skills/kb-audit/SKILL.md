# KB-Audit · 知识库健康检查与审计

## 用途

将 `[[Agent标准作业流程]]` Agent 9（健康检查）中约 500 字的手动检查清单封装为 **一句话触发** 的自动化流程。Agent 只需说"健康检查"，skill 自动完成全库扫描、问题分类、自动修复、报告输出的全链路。

**核心价值**：
- 避免 Agent 每次手动逐项核对 frontmatter 字段
- 自动检测模板版本滞后、status 与内容不匹配等系统性问题
- 低风险问题（如缺失 updated、template_version 旧版）自动修复
- 输出结构化报告，P0/P1/P2 分级清晰

---

## 触发条件（满足任一即激活）

| 场景 | 示例用户消息 |
|:---|:---|
| **主动审计** | "健康检查" / "扫一遍全库" / "审计一下" |
| **定期维护** | "月度巡检" / "全库体检" / "跑一下 audit" |
| **批量修改后** | "我刚改了一批模板，检查下有没有问题" |
| **质量担忧** | "最近建的 KP 质量怎么样" / "有没有格式问题" |

## 执行步骤

### Step 1: 全库扫描（~10-20 秒）

收集 `03-知识点/` 下所有 `.md` 文件，提取 frontmatter。

```
扫描范围：03-知识点/（排除 .obsidian/、.trash/）
提取内容：YAML frontmatter + 正文前 100 字（用于 status 匹配判断）
```

建立索引：
- `kpFiles`：所有知识点文件路径 + frontmatter 字典
- `templateVersion`：当前模板版本（从 `11-模板/模板-知识点.md` 读取 `template_version`）

### Step 2: 七维度检查（~20-30 秒）

对每个 KP 执行以下检查：

#### A. frontmatter 必填字段完整性

**必填字段**（所有 KP 必须有）：
`title`, `type`, `subject`, `module`, `status`, `updated`, `tags`

**建议字段**（已填充/初稿状态应有）：
`syllabus_code`, `parent_module`, `difficulty`, `importance`, `sources`

**图片字段**（必须成组存在）：
`has_images`, `image_count`, `images_priority`, `images_note`

| 检查结果 | 严重程度 |
|:---|:---:|
| 缺少 `title` / `type` / `status` | P0（阻塞） |
| 缺少 `subject` / `module` / `updated` | P1（高优） |
| 缺少 `syllabus_code` / `difficulty` / `importance` | P2（建议补） |
| 图片四字段不完整（如缺 `images_note`） | P1（高优） |

#### B. 模板版本一致性

读取 `[[11-模板/模板-知识点]]` 的 `template_version` 作为基准。

| 检查结果 | 严重程度 | 处理方式 |
|:---|:---:|:---|
| `template_version` 缺失 | P1 | 自动填入当前版本 |
| `template_version` < 当前版本 | P2 | 报告清单，建议升级 |
| `template_version` > 当前版本 | P0 | 模板可能未同步，警告 |

#### C. status 与内容匹配度

| status 值 | 预期正文特征 | 检查规则 |
|:---|:---|:---|
| `已填充` | 正文 > 500 字，有多节内容 | 正文 < 300 字 → P1（疑似应为"初稿"） |
| `初稿` | 正文 100-500 字，核心内容有但待完善 | 正文 > 800 字 → P2（可考虑升"已填充"） |
| `骨架` | 正文 < 100 字，只有标题和占位符 | 正文 > 200 字 → P1（疑似应为"初稿"） |
| `框架` | 由 Agent 自动创建的待填充骨架 | 正文有实质内容 → P2（可考虑升级） |

> ⚠️ **安全原则**：只标记疑似 mismatch，**不自动修改 status**。status 变更需用户确认。

#### D. 图片字段合规性

根据 SOP 规则 7 的图片字段语义：

| 字段组合 | 预期状态 | 异常检查 |
|:---|:---|:---|
| `has_images: true` + `image_count: 0` | 矛盾 | P1（声称有图但计数为0） |
| `has_images: true` + `images_note` 为空 | 未标注 | P2（应写"配了什么图"） |
| `has_images: false` + `images_priority: high` | 矛盾 | P1（声称不需图但标高优） |
| `images_note` 为空 | 未评估 | P2（需主动评估并写理由） |

#### E. 考纲关联完整性

| 检查项 | 规则 | 严重程度 |
|:---|:---|:---:|
| `syllabus_code` 为空且 status = "已填充" | 已填充 KP 应有考纲编号 | P2 |
| `parent_module` 为空 | 应指向所属模块总览 | P2 |
| `syllabus_code` 格式异常 | 应为数组如 `["6"]` 或 `[6]` | P1 |

#### F. 时间戳与时效

| 检查项 | 规则 | 严重程度 |
|:---|:---|:---:|
| `updated` 缺失 | 所有 KP 必须有更新日期 | P1 |
| `updated` 格式异常 | 应为 `YYYY-MM-DD` | P1 |
| `updated` > 90 天前且 status != "已填充" | 长期未更新 | P2（建议复查） |
| `updated` > 180 天前且 status = "已填充" | 可能需要内容刷新 | P2 |

#### G. 链接健康（引用自 kb-link-doctor）

复用 kb-link-doctor 的扫描结果（如有最近报告）：
- 若 24 小时内有断链检测报告，直接引用其统计
- 若无，快速扫描 KP 文件中的 wikilink，统计断链数（不做详细分类，只给总数）

### Step 3: 自动修复（低风险问题）

以下问题**直接 Edit 修复**，无需用户确认：

| 问题 | 修复方式 |
|:---|:---|
| 缺少 `updated` | 填入今天日期 `YYYY-MM-DD` |
| 缺少 `template_version` | 填入当前模板版本 |
| `updated` 格式错误 | 标准化为 `YYYY-MM-DD` |
| `tags` 缺失或为空 | 填入 `[化竞]` |
| `aliases` 缺失 | 填入 `[]` |
| `key_images` 缺失 | 填入 `[]` |

> ⚠️ **安全原则**：不修改 `status`、`title`、`syllabus_code` 等语义字段。只修复格式/缺省值问题。

### Step 4: 问题分级与输出报告（~10 秒）

```markdown

| 指标 | 数值 | 状态 |

### 二、P0 阻塞问题（需立即处理）

| 文件 | 问题 | 建议 |

| 文件 | 问题 | 建议 |
|:---|:---|:---|
```

- 报告保存到 `09-审计报告/YYYY-MM-DD-健康检查.md`
- 或追加到 `00-首页/工作日志/YYYY-MM-DD.md`
- 更新 `[[状态摘要]]` 中的全局状态描述（若审计发现结构性缺口）

## 特殊场景处理

### 场景 A：指定目录审计

用户说"只检查有机化学的 KP"。

处理：缩小扫描范围到 `03-知识点/有机化学/`，但**模板版本基准仍需全库统一读取**。

### 场景 B：模板版本升级后审计

用户说"我刚把模板从 v1.3 升级到 v1.4，检查下影响范围"。

处理：
1. 执行常规审计
2. 额外输出"模板版本滞后"专项统计
3. 若滞后文件数 > 20，建议分批升级，不要一次性全部修改

### 场景 C：问题数异常激增

如果本次 P0+P1 问题数比上次增加 >50%：
- 输出 ⚠️ 警告："问题数异常增加，可能由批量导入/模板变更导致"
- 建议启用 SOP 完整模式，逐项核查后再批量修复

## 与 SOP 的衔接

```
用户说"健康检查"
    ↓
kb-audit Skill 触发
    ↓
自动扫描 → 七维度检查 → 自动修复低风险问题
    ↓
输出报告（含 P0/P1/P2 分级清单）
    ↓
用户对 P0/P1 给出决策 → Agent 执行修复
    ↓
Phase 4 归档：报告存入 09-审计报告/ + 更新工作日志
```

**关键原则**：
- Skill 只做"检测 + 格式级自动修复"，不做内容级修改（如 status 变更需用户确认）
- 任何涉及 >5 个文件的 status 修改，必须在报告中明确列出并建议用户确认
- 修复后的 frontmatter 必须保持 YAML 格式正确（缩进、数组语法）

## 维护说明

- **版本**：v1.0（基于 SOP v1.4.4 + KP 模板 v1.3）
- **更新时机**：
  - KP 模板 frontmatter 字段有变更时
  - 新增/删除必填字段时
  - status 值列表变更时
- **测试方式**：
  - "健康检查" → 应输出报告，P0 率 <1%
  - 故意写一个缺少 `updated` 的 KP → 应被自动修复
  - 验证 `status: 已填充` + 正文 < 100 字 → 应标记为 P1 mismatch
