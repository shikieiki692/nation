# Agent最小执行协议

> **用途**：给 Agent 一份日常默认执行的最小规则集。
> 目标不是覆盖全部细节，而是让 Agent 在**更听话、更省 token**的前提下，稳定完成”路由 → 执行 → 收工同步”闭环。

> **默认原则**：普通任务先读本页，按本页 + 对应轻量提示词执行；只有复杂、高风险、跨模块或多轮失败任务，才升级到 [[Agent标准作业流程]] 和长版提示词。

## 一、任务驱动执行流程（默认模式）

> **核心变更**：不再”先读一堆文件再开工”。改为”先建任务卡 → `task_type` 驱动一切”。任务卡是你手里的锚点，捏着它就不会漂移。

```text
用户指令
  ↓
[STEP -1] 开工前健康检查
  检查 [[质量看板]] → 若有大量 error/warning 先排查再开工
  若上一轮未运行 validate_kb.py 收工验证 → 当前先补跑再开工
  ↓
[STEP 0] 检查会话交接笔记
  若有未读的交接笔记 → 优先读它，跳过未变更的入口文件
  若无 → 继续
  ↓
[STEP 1] 快速定位
  读 [[状态摘要]] → 当前状态与风险
  读 [[活跃任务]] → 健康检查 + 现有 active 卡
  跳过 [[Agent战略方向]]（除非用户没指定任务）
  ↓
[STEP 2] 明确任务 → 创建/认领任务卡
  当前任务是否有对应的现有任务卡？
  ├─ YES → 确认 status=active，补 task_type，claim it
  └─ NO  → 新建任务卡（用 [[11-模板/模板-活跃任务卡]]）
  
  关键：必须在 Phase 1 结束前填好 task_type 字段
  ↓
[STEP 3] task_type 路由 → 决定”读什么 + 用什么提示词 + 收工同步哪”
  查下方”任务类型路由表”，不要自己猜
  
  🔍 内容修改前强制检查：若涉及学生讲义内容充实/修改，
  必须先读对应备课大纲或备课框架中的**深度边界**（depth boundary）
  确认修改范围在第一轮边界内，避免越界添加第三轮内容
  ↓
[STEP 3.5] 输出执行声明（约束锚点）
  在 Phase 1 结束时，输出以下声明。这是承诺装置，写完就不会忘：
  
  ```
  [执行声明]
  ├─ task_type: {类型}
  ├─ 开工前必读: ✅ {已读文件列表}
  ├─ 影响范围: {将改动文件}
  ├─ 模板: {模板名 / 无}
  ├─ 收工必同步: {路由表对应列}
  └─ 会话交接: 将写入 00-首页/会话交接/YYYY-MM-DD-任务名.md
  ```
  ↓
[STEP 4] 开工
  执行期间以任务卡 frontmatter 为锚点：
  - 每隔几步扫一眼 status——是否还 active？
  - 若发现需要改方向 → 先更新任务卡，再继续
  ↓
[STEP 4→5 阶段检查点] 在 Phase 2→3（执行→验证）和 Phase 3→4（验证→归档）之间各输出检查点。每个检查点使用以下模板：

  **Phase 2→3（执行→验证）**：
  ```
  [Phase 2→3]
  ├─ 已完成:
  │  · {已完成的改动}
  ├─ 实际修改文件: {文件数}
  └─ 待验证:
     [✅] {验证项1}
     [  ] {验证项2}
  ```

  **Phase 3→4（验证→归档）**：
  ```
  [Phase 3→4]
  ├─ 验证结果: ✅ {通过数}/{总数} 通过
  ├─ 未通过: {未通过项 / 无}
  ├─ 待同步:
     [✅] 工作日志已更新
     [✅] 任务卡 evidence 已填
     [  ] 会话交接笔记已写
     [  ] 状态摘要（按 task_type 判断是否需要更新）
  ```
  ↓
[STEP 5] 收工 → 先更新任务卡，再同步
  1. 先写 [[工作日志]]（过程记录）
  2. 更新任务卡：status / completed / evidence
  3. 由 task_type 决定还需同步哪里（见下方”路由表”）
  4. 写 [[00-首页/会话交接/]] 笔记（下一位 Agent 用）
  ↓
[STEP 6] 写会话交接笔记
  15 行内写完：本轮做了什么、改了哪些文件、下一轮建议
  存在 `00-首页/会话交接/YYYY-MM-DD-任务名.md`
```

### 约束规则

| 规则 | 说明 | 违反后果 |
|:---|:---|---:|
| **开工前必须有卡** | 没有对应任务卡 → 先建卡再动手。建卡 = 承诺”我要做这件事” | 路由表无入口，Agent 无法确定提示词和同步目标 |
| **task_type 必填** | 缺 task_type 的任务卡 = 无效卡，路由系统不知道送你到哪 | 任务调度中断，自动汇总无法归类 |
| **执行声明必输出** | Phase 1→2 前输出 `[执行声明]`。写出来就不会忘 | Agent 中途漂移，收工时忘记同步目标 |
| **阶段检查点必输出** | Phase 2→3 和 Phase 3→4 之间各输出 1 行检查点 | 验证遗漏，错误扩散到下游文件 |
| **evidence 不可为空** | Phase 4 不填 evidence = 任务未完成，下一位 Agent 应打回 | 自动汇总报异常，下一位 Agent 无法判断是否真正完成 |
| **深度边界先查** | 修改Round N讲义前先读对应备课框架的深度边界，确认不越界添加后续轮次内容 | 内容超纲，学生跟不上，需用户纠正后才回退 |
| **status 反映真实状态** | 做完了必须改 completed，被阻塞必须改 blocked | 活跃任务聚合失准，状态摘要误导后续判断 |
| **先修卡再修入口页** | 若活跃任务和状态摘要冲突 → 以任务卡状态为准，修完卡再考虑是否更新摘要 | 双重真相源冲突，自动汇总溢出冗余数据 |
| **会话交接笔记必写** | 下一位 Agent 靠它恢复上下文。不写 = 下一次从头读全部入口文件 | 下一轮 Agent 需重读全部入口文件，浪费 token |

## 二、提示词读取策略（省 token 版）

### 默认

- **先用轻量提示词**：`11-模板/Agent-*-轻量.md`
- **长版提示词** `[[Agent提示词-知识库构建]] / [[Agent提示词-知识库维护]] / [[Agent提示词-知识库使用]]` 默认只在以下情况补读：

1. 本次任务有多个子模式，轻量版不足以判断边界
2. 任务涉及批量操作、迁移、合并、弃用、审计
3. 任务涉及 Agent 1A / 1B 等特殊子流程
4. 前一次尝试已经失败，怀疑遗漏了细则
5. 你需要查某个字段、表格、规则的精确定义

### 完整 SOP 升级条件

只有满足以下任一条件，才从本页升级到 [[Agent标准作业流程]]：

1. 跨两个及以上模块，且你仍不确定上下游和收工回写
2. 会修改 SOP、模板、任务卡规则、系统入口文件
3. 批量改动文件较多，或涉及迁移/合并/弃用
4. 任务存在较高误改风险，需严格执行 Phase 3-4
5. 你已经出现一次明显跑偏、漏同步或规则冲突

## 五、收工清单（任务卡驱动版）

> **一句话**：收工先更新任务卡 → `task_type` 自动决定同步哪里。不需要再查矩阵。

### 固定必须做的

| 顺序 | 动作 | 说明 |
|:---:|:---|---|
| ① | 写 [[工作日志]] | 过程记录 + 验证结果 + 证据链接 |
| ② | 自动化验证 | `python 11-模板/scripts/validate_kb.py --changed <改动文件> ...`<br>确认无新增 error/warning 后再继续 |
| ③ | 更新任务卡 | `status` / `completed` / `evidence` 必须填 |
| ④ | 查路由表同步 | 按 `task_type` 查上方"任务类型路由表"的"收工必须同步"列 |
| ⑤ | 写会话交接笔记 | `00-首页/会话交接/YYYY-MM-DD-任务名.md`，15 行内 |

### 📋 收工归档 Checklist（标准版）

> 用于"按之前的工作流存个档"场景。完成以下 5 步即可，不必逐个文件判断。

| # | 动作 | 涉及文件 | 完成标记 |
|:---:|:---|---|:---:|
| 1 | **更新证据字段** | 对应任务卡 → 补 `evidence` | [ ] 1 |
| 2 | **写工作日志** | `00-首页/工作日志/YYYY-MM-DD.md` → 记录修改清单+验证结果+决策依据 | [ ] 2 |
| 3 | **更新活跃任务判断** | [[活跃任务]] → 最顶部 `✅ 当前判断` 块 | [ ] 3 |
| 4 | **验证 frontmatter 完整** | 确认改动文件的 `updated`/`image_count`/`stage` 等字段已更新 | [ ] 4 |
| 5 | **检查断链** | `validate_kb.py --changed <files>` 确认无新增断链 | [ ] 5 |

### 入口页同步判断（仅以下情况需手动）

| 结论变了？ | 必须同步 |
|---|---|
| 阶段 / 阻塞 / 下一步 | [[状态摘要]] |
| 长期优先级 / 待用户决策 | [[Agent战略方向]] |
| 有新坑可复用 | [[经验沉淀]] |

> **默认**：如果 [[自动汇总-异常巡检]] 和 [[自动汇总-任务卡健康检查]] 全部绿灯，状态摘要和战略方向不需要手动更新。只有阶段判断、阻塞项、长期方向变了才需要动入口页。

### 经验沉淀判定

符合以下任一就写，否则只写日志就够了：
1. 同类坑以后大概率还会再踩
2. 不是单文件特例，而是工作流级问题
3. 这条规避法能直接节省后续 Agent 的时间

## 六、完整工作链检查

一条完整的工作链应能回答这 6 个问题。答不上来就不要硬做，先升级读取层级：

1. **有卡吗？** → 任务卡建了没？`task_type` 填了没？
2. **声明了吗？** → `[执行声明]` 输出了没？收工必同步列出来了吗？
3. **路由对吗？** → `task_type` 对应的提示词和模板找对了没？
4. **开工前该读的读了吗？** → 路由表里"开工前必读"列的文件读了没？
5. **收工该同步的同步了吗？** → task_type 路由表"收工必须同步"列做了没？evidence 填了没？
6. **会话交接写了吗？** → 下一位 Agent 能靠你的笔记恢复上下文吗？
