### 三、工作流问题复盘

本轮发现的核心问题不是“没人记得关卡”，而是以下三类混在了一起：

1. **可执行任务**：例如目录建立、专题产出、脚本改造
2. **可由审计结论关闭的任务**：例如“接受旧命名”“改为复用已有专题页”
3. **待外部决策 / 待真实试点的任务**：例如实验模块、纸质打印验证、学生反馈闭环

如果三类都默认写成 `active`，任务池会逐渐失真。

进一步看，Agent 低效通常出在两处：

1. **开工前不知道这次任务会牵动哪几层**，于是读了一圈还是没抓住真相源
2. **收工时只记得写工作日志，不记得回写系统层**，于是 `状态摘要 / 活跃任务 / 备课待办 / 战略方向` 很快漂移

### 四、当前建议

1. 能立即执行的前置项继续保留 `active`
2. 明确等待用户、实体打印、课堂反馈的卡，优先改为 `blocked`
3. 任何日志里出现“已完成 / 全部闭环 / 改为复用 / 正式接受旧命名”时，同步更新任务卡 frontmatter，不把收口信息只留在正文或审计表里

## 📅 2026-06-07 提示词瘦身 + 最小执行协议接入

**任务**：检查现有提示词和工作流架构是否完整，并在不牺牲约束力的前提下压缩默认读取成本，让 Agent 更少读长文、更少漏同步。

- 将三份长版提示词：
  - [[11-模板/Agent提示词-知识库构建]]
  - [[11-模板/Agent提示词-知识库维护]]
  - [[11-模板/Agent提示词-知识库使用]]
  的顶部口径统一为：**默认先读最小协议 + 对应轻量提示词，复杂场景再升级完整 SOP / 长版提示词**
- 在 [[Agent标准作业流程]] 增补“轻量执行口径”，明确完整 SOP 的升级条件
- 在 [[Agent任务分解指南]]、[[11-模板/模板]]、[[首页]]、[[Agent操作与知识库导航说明]] 接入新入口，避免轻量提示词存在但入口层不调用

### 二、这轮识别出的结构性问题

1. **长版规则齐全，但默认入口过重**：提示词长期要求“先读完整 SOP”，会让普通任务也携带高上下文成本
2. **轻量提示词已存在，但未成为默认入口**：导致“有瘦身材料，但系统没真正切换”
3. **问题处理链不够短**：此前虽有 `经验沉淀` 和 `工作日志`，但缺少一句话级别的“先查哪里、什么时候升级”的协议
4. **收工分层已大体齐全，但缺少最短版判断表**：现在已通过最小协议把 `工作日志 / 活跃任务 / 状态摘要 / 战略方向 / 复盘记录 / 经验沉淀` 的分工重新压缩成可执行表

### 三、当前结论

- **工作流架构总体是完整的**：从用户指令 → 路由 → Agent → 模板 → 执行 → 问题查阅 → 收工同步 → 经验沉淀，这条链路各层都有落点
- **此前主要问题不是“缺文件”，而是“默认路径过长、升级条件不够显式”**
- 因此本轮优化重点不是继续加很多新文档，而是把现有文档之间的**默认调用顺序**改对
