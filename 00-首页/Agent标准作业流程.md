# Agent 标准作业流程（SOP）

> **本文件是知识库 Agent 的"飞行检查表"。**
> 任何 涉及到知识库内容的Agent 在执行任务之前、过程中、之后，都必须按照本文档定义的协议走完闭环。
> 无论任务大小（提炼一章 vs 修改一处 typo），都建议至少快速跳过 Session 初始化、ASSESSING、CLOSING。

> **v1.5.2 新口径**：日常任务默认先读 [[Agent最小执行协议]] + 对应轻量提示词；本文件作为**完整版规则书**，在复杂、跨模块、批量、高风险、工作流改造场景中升级使用。见 §四 升级条件。

---

## ⚡ 速查卡（Agent 30 秒版）

| Phase | 核心动作 | 检查问句 |
|:---:|:---|:---|
| ASSESSING | 读 [[状态摘要]] + [[活跃任务]] + [[Agent战略方向]]（默认）或 8 文件完整模式；跨模块时补读 [[Agent模块关系与收工同步清单]] | 清楚库存、方向、任务边界、模板位置、上下游模块？ |
| PLANNING | 识别任务类型 → 选 Agent → 与用户确认计划 | 计划清楚到用户可审阅？ |
| EXECUTING | 按提示词执行，批量 ≤20/Agent | 源文件未编造、异常有标注？ |
| VERIFYING | 抽查 3–5 个产出，验 frontmatter + 链接 | 读了 3+ 样本、对照了源文件？ |
| CLOSING | 写工作日志 → 更索引 → 同步入口文件 → 汇报用户 | 日志写了、⚠️项记录了、入口状态同步了、总结给了？ |

**红线**：相对路径 | 不编造 | 未审核不进正式区 | 批量改前问用户

## 🧭 入口路由（v1.2 新增）

> 收到用户消息后，**第一步是判断意图归哪一类**，再决定走哪条路径。

```
用户消息 → 判断意图 → 选定路径

  📥 构建（提炼/填充/拆题/链接/审查）   → 走 完整生命周期 + [[Agent提示词-知识库构建]] 1-6
  🔧 维护（优化/合并/弃用/健康检查）    → 走 完整生命周期 + [[Agent提示词-知识库维护]] 7-9
  📚 使用（备课/答疑）                  → 走 完整生命周期 + [[Agent提示词-知识库使用]] 10-11
  💬 提问（基于 KB 直接答）             → 不走 EXECUTING 阶段到 CLOSING 阶段，直接基于知识库内容回答 + 引用 wikilink
  🛠️ 元讨论（改 SOP/模板/重构）         → 走完整 完整生命周期，特别注意 PLANNING 用户确认
  🪶 一般问题（与 KB 无关）             → 直接回答即可，不进 SOP
```

### 判断要点

| 用户用词 | 路径 |
|---|---|
| "提炼"/"填充"/"创建"/"建知识点" | 📥 构建 |
| "优化"/"改进"/"完善"/"合并"/"弃用"/"扫一遍" | 🔧 维护 |
| "备课"/"做大纲"/"准备一节课"/"答疑" | 📚 使用 |
| "什么是 X"/"X 怎么反应"/"为什么 X" | 💬 提问 |
| "改 SOP"/"调整模板"/"重新设计" | 🛠️ 元讨论 |
| 与化学/KB 无关的闲聊或简单技术问题 | 🪶 一般 |

```
                              ↓

                              ↓
┌──────────────────────────────────────────────────────────────────┐
│  EXECUTING  执行                                                    │
│  ─ 按提示词执行 / 启动并行 Agent / 异常优雅降级                      │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────────────────────────────────────────────────┐
│  VERIFYING  验证 & 修正                                             │
│  ─ 抽查 3-5 个产出 → 校验 frontmatter → 校验链接 → 必要时修正        │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────────────────────────────────────────────────┐
│  CLOSING  记录 & 归档                                             │
```

# 一、Session 初始化协议

> **目的**：在动手之前，让 Agent 对"知识库当前状态 + 任务边界"有清晰认识，避免重复劳动、错位归档、考纲偏离。
> **v1.5.3 口径统一**：默认读取 3 个文件（[[状态摘要]] + [[活跃任务]] + [[Agent战略方向]]），仅在特殊情况启用完整模式。若 `[[活跃任务]]` 健康检查暴露异常，先修任务卡，再做优先级判断。

> **v2.0 前置步骤**：ASSESSING 开始前，先执行 [[Agent最小执行协议#一、任务驱动执行流程（默认模式）]] 的 STEP 0-2（检查会话交接 → 快速定位 → 创建/认领任务卡）。以下流程为衔接步骤。

## 1.1 上下文加载（读三入口文件）

### 默认模式（速读三文件）⭐

| 顺序 | 文件 | 读取目的 |
|:---:|------|---------|
| 1 | `[[00-首页/状态摘要]]` | 会话快照：库存现状 / 最近 3 次任务 / 高优 Follow-up / 踩坑速查 / 体系版本 / 入口路由 |
| 2 | `[[00-首页/活跃任务]]` | 执行聚合：当前未完成任务、blocked、近期完成、任务卡健康检查；真相源在 `00-首页/活跃任务/` |
| 3 | `[[00-首页/Agent战略方向]]` | 长期方向：当前阶段判断 / 工作优先级 / 用户未明确任务时的默认推进方向 |

> ⚡ **节省 ASSESSING 成本，同时保留方向感与执行真相源**。`状态摘要` 管会话快照，`活跃任务` 管执行聚合与任务卡健康检查，`Agent战略方向` 管长期优先级。三者关系速览见 [[Agent入口文件关系说明]]。

### 模块关系补充卡（跨模块任务必读）⭐

当任务满足以下任一条件时，默认模式之后立刻补读 [[Agent模块关系与收工同步清单]]：

1. 用户一句话同时涉及 **资料 / KP / 专题 / 课件 / 学生侧材料** 中的两层及以上；
2. 你不确定"该先读哪个模块、做完后要回写哪里"；
3. 本次任务可能改变 `状态摘要` 的"当前重心 / 阻塞 / 下一步"；
4. 本次任务可能让"审计结论吸收旧任务"，需要关闭或阻塞任务卡。

> **目的**：用 90 秒把"模块关系 + 收工同步矩阵"理清，避免做对内容但忘了回写系统层。

### 完整模式（读全部 8 个文件）

> 仅在以下情况启用：
> ① 跨模块大批量任务（如 APiIC 全书提炼）
> ② SOP 有版本变更
> ③ 用户明确要求"完整加载上下文"
> ④ 状态摘要中找不到所需信息

| 顺序 | 文件 | 读取目的 |
|:---:|------|---------|
| 1 | `[[00-首页/状态摘要]]` | 总览（同默认模式） |
| 2 | `[[00-首页/活跃任务]]` | 当前执行任务总表（由任务卡聚合）+ 健康检查 |
| 3 | `[[00-首页/Agent战略方向]]` | 当前阶段判断 / 方向优先级 / 默认行为 |
| 4 | `[[00-首页/工作日志]]` | 最近详细产出与踩坑（注意：内容较长） |
| 5 | `[[00-首页/经验沉淀]]` | 完整版踩坑库（8 大类） |
| 6 | `[[01-考纲导航/中国化学奥林匹克基本要求-总览]]` | 考纲条目对照 |
| 7 | `[[00-首页/知识库待补充清单]]` | 历史缺口闭环与监控（仅在需要追溯建设期缺口或核查监控结果时读） |
| 8 | `[[待建KP优先级清单]]` | 真题高频 REDLINK（资料导入时必查） |

## 1.2 检查点探测与恢复

> 若 `[[活跃任务]]` 健康检查暴露异常（如缺字段 / completed 漏日期），**先修任务卡，再做优先级判断**。不要跳过健康检查直接推进新任务。

## 1.3 健康检查优先

### Phase 0 检查问句（自检）

- [ ] 我是否清楚知识库现在有什么、缺什么？
- [ ] 我是否看过 `[[活跃任务]]` 的健康检查结果？若有缺字段 / completed 漏日期，是否先修再判断任务优先级？
- [ ] 我是否清楚当前阶段优先级、默认推进方向和不该做什么（见 `[[Agent战略方向]]`）？
- [ ] 我是否知道这个任务对应的考纲条目编号（如适用）？
- [ ] 我是否找到了对应的模板？
- [ ] 我是否找到了 1-2 个"标杆产出"作为格式参考？
- [ ] **任务卡已建？`task_type` 已填？**（见 [[Agent最小执行协议#一、任务驱动执行流程（默认模式）]]）
- [ ] 我是否知道这个任务在工作日志里有没有相关上下文（之前做过类似的吗）？

> ⚠️ **如果任何一项答不上来，立即回到 Session 初始化**，不要"先做起来再说"。
> ⚠️ **如果默认模式信息不够，立即升级到完整模式**，不要靠猜。

## 1.4 已读注册 + 指纹比对

> 在上下文加载完成后，Agent 应记录本次已读文件清单，用于后续阶段的增量比对（如 VERIFYING 验证时确认"是否读过源文件"）。指纹比对机制参见 [[经验沉淀]]。

# 二、Task 生命周期协议

## 2.1 ASSESSING 阶段

> **目的**：将自然语言任务映射到具体的 Agent 提示词、文件路径、批次策略，并向用户呈现可审阅的执行计划。

### Step 1：任务类型识别

将用户请求映射到 **11 个标准 Agent**（v1.2 扩展，按用途分 3 大类）。若任务跨越多个类型，拆分为子任务并按依赖顺序执行。

#### 📥 构建型（创造新内容） · [[Agent提示词-知识库构建]]

| 用户描述关键词 | 对应 Agent | 提示词位置 |
|:---:|:---:|:---:|
| "提炼某章 / 转结构化 / 从教材提取" | 1. 资料提炼 Agent（含子模式 1A 网课思路 / 1B 资产回流） | `[[Agent提示词-知识库构建#1. 资料提炼 Agent]]` |
| "填充骨架 / 把知识点写完整 / 从提炼笔记补内容" | 2. 知识点填充 Agent | `[[Agent提示词-知识库构建#2. 知识点填充 Agent]]` |
| "提取习题 / 拆解习题册" | 3. 习题提炼 Agent | `[[Agent提示词-知识库构建#3. 习题提炼 Agent]]` |
| "拆题 / 单题入库 / 题库归档" | 4. 题库拆分 Agent | `[[Agent提示词-知识库构建#4. 题库拆分 Agent]]` |
| "建立链接 / 双向引用 / 关联知识点和题目" | 5. 交叉链接 Agent | `[[Agent提示词-知识库构建#5. 交叉链接 Agent]]` |
| "审查 / 检查 / 找问题"（针对新产出） | 6. 质量审查 Agent | `[[Agent提示词-知识库构建#6. 质量审查 Agent]]` |

#### 🔧 维护型（优化已有内容） · [[Agent提示词-知识库维护]]

| 用户描述关键词 | 对应 Agent | 提示词位置 |
|:---:|:---:|:---:|
| "优化 / 改进 / 完善某 KP / 这个写得太浅" | 7. KP 优化 Agent | `[[Agent提示词-知识库维护#Agent 7 · KP 优化 Agent]]` |
| "合并 / 去重 / 弃用 / 这俩是同一个" | 8. KP 合并/弃用 Agent | `[[Agent提示词-知识库维护#Agent 8 · KP 合并/弃用 Agent]]` |
| "健康检查 / 审计 / 扫一遍 / 全库巡检" | 9. 健康检查 Agent | `[[Agent提示词-知识库维护#Agent 9 · 健康检查 Agent]]` |

#### 📚 使用型（输出教学物料） · [[Agent提示词-知识库使用]]

| 用户描述关键词 | 对应 Agent | 提示词位置 |
|:---:|:---:|:---:|
| "备课 / 做大纲 / 准备一节课 / 整理讲义" | 10. 备课 Agent | `[[Agent提示词-知识库使用#Agent 10 · 备课 Agent]]` |
| "回答 / 答疑 / 学生问 X" | 11. 答疑 Agent（v0.5 占位） | `[[Agent提示词-知识库使用#Agent 11 · 答疑 Agent（占位 v0.5）]]` |

#### 不属于以上 11 类
单独说明，临时定义工作流；优先考虑是否能用现有 Agent 组合解决。

### Step 2：前置条件核查

| 检查项 | 检查方法 | 不通过时怎么办 |
|------|--------|------------|
| 源文件是否存在 | `Glob` 或 `Read` 验证 | 让用户补充源文件路径 |
| 目标目录是否就绪 | `Glob` 验证目录存在 | 在执行计划中加入"创建目录"步骤 |
| 模板是否可用 | 读取对应模板文件 | 让用户确认是否需要新建模板 |
| 命名冲突（目标文件已存在） | `Glob` 检查目标路径 | 询问用户：覆盖 / 跳过 / 改名 |
| 考纲条目编号是否明确 | 与考纲总览对照 | 在计划中标注"待用户确认编号" |
| 是否有上一轮 follow-up 未处理 | 先看 `[[活跃任务]]`，必要时再翻工作日志 | 在计划末尾加入"处理 follow-up" |

### Step 3：执行计划起草

向用户呈现的执行计划应至少包含：

```
## 执行计划草案

### 任务类型
{对应的 Agent 提示词名称}

### 任务范围
- 源文件：{绝对/相对路径列表}
- 目标产出：{产出文件路径或目录 + 估计数量}
- 涉及考纲条目：{编号列表}

### 批次策略
- 单 Agent / 并行 N 个 Agent
- 每批 ~M 个文件，分 K 批

### 预期产出
- 数量：约 {N} 个文件
- 形态：{骨架 / 已填充 / 单题文件 / 其他}
- 关键字段：{需要重点保证的 frontmatter 字段}

### 已知风险 / 不确定点
- {例如：源文件中第 X 章疑似超纲，是否提取需用户确认}
- {例如：与现有 [[XYZ]] 知识点存在重叠，是合并还是新建}
- {例如：范围内 KP 过多，本轮仅先处理主链桥梁页，枝节页后置}

### 验证方案
- 抽查样本数：{3-5}
- 验证维度：{格式 / frontmatter / wiki-link / 内容准确性}
```

### Step 4：与用户确认

> 除非用户明确说"放手做"，否则在向并行 Agent 发指令、批量写文件、修改既有内容之前，先把执行计划告诉用户并等待确认。

确认环节避免的反模式：
- ❌ "我先动手做几个看看效果" — 一旦方向偏了就要返工大量内容。
- ❌ "我猜用户想要 X 格式" — 直接问比猜更高效。
- ❌ "我把所有 100 个文件都改了" — 应分批，每批确认后再继续。

### ASSESSING 检查问句（自检）

- [ ] 我选定的 Agent 提示词是否真的匹配任务？
- [ ] 前置条件是否全部满足？
- [ ] 执行计划是否清楚到"用户读完就知道会产生什么"？
- [ ] 我是否标出了风险与不确定点（哪怕只是"这一点我不确定"）？
- [ ] 我是否给出了"出错时如何回滚 / 跳过"的策略？
- [ ] **若涉及多套网课/教材资料**：我是否已读 `[[00-首页/方法论-多源教学资料整合]]`？是否清楚"顺序自己定、传授借鉴人、资产统一存"的原则？
- [ ] **若涉及两个及以上模块**：我是否已读 `[[Agent模块关系与收工同步清单]]`？是否知道上游来源、下游影响和收工回写位置？
- [ ] **若涉及题库筛题 / 真题链 / 讲后练**：我是否先看了 `[[02-数据库/数据库-真题]]`，而不是直接翻题库目录？
- [ ] **若涉及备课进度判断**：我是否先看了 `[[备课思路/备课与教学思路待办]]`，而不是直接用目录猜覆盖状态？
- [ ] **若涉及第一轮基础班新授课治理**：我是否先区分了 `现役 / 历史 / 待处理`？是否已查看 `[[04-课件/新授课/2026-06-15-第一轮基础班课堂执行页总览]]` 与 `[[04-课件/新授课/2026-06-15-第一轮基础班现役历史待处理清单]]`？
- [ ] **若涉及备课 / 讲义 / PPT / DOCX 加工**：我是否先明确了知识边界、课时数、班型深度、主教材范围、可借鉴网课范围，以及三类交付物各自的职责？
- [ ] **若涉及备课**：我是否真的把 `KP` 当成可抽料资产，而不是只把 `referenced_kps` 当作挂名字段？我是否能说出哪些 `KP` 分别贡献了定义、机制、例题、易错点、图景？
- [ ] **若涉及大量 KP 的备课 / 优化**：我是否先找到了"主教学链"和"桥梁 KP"，而不是准备在大量 KP 中平均用力？
- [ ] **若本轮要扩充 KB 可复用资产**：我是否只选了 1 条主链 + 最多 2 条外扩链，并准备先做少量样板页再复制？
- [ ] **若涉及第一轮基础班化学原理备课**：我是否默认以 `普通化学原理` 为边界，并仅在战略文件已允许的主题上向 `无机化学` 轻度延伸，而不是直接滑向物化深度？
- [ ] **若涉及正式 docx/pptx 交付**：我是否已读取 `[[模板-课件加工页]]` 与 `[[模板-课件产出规范]]`，并预留 render 检查步骤？
- [ ] **若涉及正式 PPT 交付**：我是否已把"中文标题黑体、中文正文宋体（必要时黑体）、英文/数字/变量/分子式 `Times New Roman`"当成硬约束，而不是默认混排？
- [ ] **若涉及正式 PPT 交付**：我是否单独检查了教师提示框、页序建议、底部提醒条、窄栏侧注等"小字号投屏敏感区"？若宋体偏虚，是否先缩句、再加字号、必要时切黑体？
- [ ] **若涉及正式 PPT 交付**：我是否预留了"公式专项目检"，确认字符间距、上下标、化学式下标、电荷上标自然？

> ⚠️ **如果任何一项答不上来，立即回到 Session 初始化**，不要"先做起来再说"。

## 2.2 PLANNING 阶段

> **目的**：确定执行策略，适应任务规模，必要时设置确认栅栏。

### 策略选择

根据 ASSESSING 阶段选定的 Agent 提示词，确定具体执行策略（单 Agent / 并行 / 分批）。参见对应提示词中的执行指南。

### 规模适应

- 小任务（≤3 文件）：可走快速通道，跳过详细计划
- 中任务（4-20 文件）：标准计划 + 分批执行
- 大任务（>20 文件）：详细计划 + 并行 Agent + 分批确认

### 确认栅栏

在向并行 Agent 发指令、批量写文件、修改既有内容之前，先把执行计划告诉用户并等待确认（见 ASSESSING §与用户确认）。

### 备课 / KP 资产化的最小推进法（v1.5.6 新增）

当目标是"让很多 KP 真正进入备课产出"时，不按目录平均推进，而按下述顺序执行：

1. 先定 **主教学链**
   - 例如：`量子数 → 原子轨道 → 电子排布 → 元素周期律 → 电离能`
2. 再定 **外扩支链**
   - 每轮最多 1~2 条，如：
   - `电离能 → 电子亲合能 → 电负性`
   - `VSEPR → 杂化轨道理论 → 分子轨道理论`
3. 每条链先补 **桥梁 KP**
   - 能承接上下游
   - 能讲机制
   - 能落例题 / 易错点 / 图景
4. 每轮只要求产出 **2~5 张可抽料样板页**
   - 包含 `lesson_asset_ready`
   - 包含 `lesson_asset_profile`
   - 包含完整"备课可抽料资产"
5. 样板跑通后，再向旁支复制

> 目标不是"本轮补很多页"，而是"本轮把一条链跑通到可以复制"。

## 2.3 EXECUTING 阶段

> **目的**：按 PLANNING 阶段的计划稳定输出，过程中保留可追溯的痕迹。

### 执行原则

1. **遵循专用提示词**
   严格按 `[[Agent提示词-知识库构建]]` 中对应 Agent 的"提示词模板"操作，不擅自增减字段。

#### 备课链路追加口径（2026-06-16）

当任务进入 `备课大纲 → 学生讲义 → PPT/DOCX` 链路时，默认执行以下顺序：

1. **先定边界，再抽料**
   - 先写清：班型、课时数、知识边界、允许延伸到哪里、哪些内容只点到为止。
   - 第一轮基础班默认以 `普通化学原理` 为主边界；可借 `无机化学` 补足直观与推导，但不要越到物理化学型证明或过深模型。

2. **先盘资源，再写内容**
   - 必查：备课框架、对应 KP / 专题、教材提炼、网课提炼、必要真题链。
   - 抽料优先级：主教材与备课大纲 > 知识库 KP / 专题 > 网课资料 > 补充题与图片资产。
   - 禁止只凭单一 Markdown 或已有 PPT 残稿直接扩写。

3. **三类交付物分工必须分开**
   - 教师稿：负责课时分配、边界、板书/PPT 分工、教学预警、例题链。
   - 学生讲义：负责完整可学内容、推导、例题、易错点、作业、自查。
   - PPT：负责投影呈现与课堂节奏，不承担"把讲义全文搬上屏幕"的任务。

4. **内容密度宁可丰富但有边界，不可空泛**
   - 第一轮材料至少覆盖：核心概念、必要推导、典型例题、易错点、与后续章节的衔接。
   - 若书上和网课稳定覆盖、且不明显超纲的内容，应优先纳入，而不是机械停在高中层面。

5. **图片与公式走课堂实用主义**
   - 图片只补"投影价值高"的教材图、结构图、分布图、实验图，不塞装饰图。
   - 公式、化学式、上下标、根号、分式必须做显示检查；不能接受 `_` 假下标、上标过远、字符留白异常直接进入正式成品。

2. **批量与并行**
   超过 ~10 个文件的任务考虑并行 Agent；批次大小建议 ~10-20 文件 / Agent。
   并行 Agent 的 prompt 必须自包含——不要假设它能看到主对话上下文。

3. **保持可追溯性**
   - 每个产出文件 frontmatter 中的 `sources` / `source_type` 字段必须填写来源；
   - 重大跳过决策（如"第 3 章因超纲跳过"）写到 CLOSING 阶段工作日志。

4. **异常优雅降级**
   遇到无法识别的内容（公式渲染失败、图片缺失、原文歧义）：
   - **不要编造**：用 `⚠️ {简短原因}` 标注；
   - **不要中断**：标注后继续处理后续内容；
   - **不要静默吞掉**：在 CLOSING 阶段的工作日志或 follow-up 清单中记录。

5. **保持原文不变**
   - LaTeX 公式、化学方程式不重写；
   - 题号不重新编号；
   - 已有内容不删除（除非用户明确要求）。

### 脏模块追踪

执行过程中发现的需要后续修正的问题（如格式不一致、链接断裂、内容缺失），记录到"脏模块清单"，在 VERIFYING 和 CLOSING 阶段统一处理。

### 动态孵化

执行过程中发现需要新建文件或模块时，按原计划的命名规范和目录结构创建，避免事后补建。

### 进度检查点

大批量任务应设置进度检查点（如每 10 个文件），暂停检查产出质量，必要时调整策略。

### 执行期常见踩坑

| 坑 | 表现 | 规避方法 |
|---|----|--------|
| 路径写错 | wikilink 显示红色 | vault 内一律用相对路径，文件名不带 `.md` 后缀 |
| YAML 数组缩进 | frontmatter 解析失败 | 数组项前留 `- `，缩进保持一致 |
| LaTeX 漏 `\mathrm` | 化学式被斜体化 | 化学元素与离子始终用 `\mathrm{}` |
| 多问小题被拆散 | (1)(2)(3) 分到不同文件 | 题库拆分时同主编号保持在一个文件 |
| 并行 Agent 重复处理 | 多个 Agent 写同一文件 | PLANNING 阶段明确每个 Agent 的文件范围（用章节或编号区段） |
| 把骨架内容当"已填充" | status 字段误改 | 只有提炼笔记真有内容才能改 status |

## 2.4 VERIFYING 阶段

> **目的**：在归档前用低成本抽查发现系统性问题，避免将错误固化到知识库。

### 验证抽查：样本选择

至少抽查 3-5 个文件，覆盖：
- **首批产出**（验证整体格式）
- **末批产出**（验证 Agent 没有"漂移"）
- **边界案例**（带图片的、特别短的、特别长的、含多问的）

### 验证维度清单

| 维度 | 检查内容 | 工具 |
|------|--------|----|
| **frontmatter 完整性** | 必填字段是否都有值（title、type、tags、updated 等） | `Read` 文件头 |
| **状态字段正确性** | `status` 是否与实际内容匹配（"骨架" vs "已填充"） | 阅读正文判断 |
| **编码完整性** | 中文正文是否无 `�` 乱码；frontmatter 中无 UTF-8 替换字符；文件实际编码为 UTF-8 无 BOM | `PowerShell` 字节扫描或 `Read` 抽查 |
| **wiki-link 有效性** | `[[...]]` 指向的文件是否真的存在 | `Glob` 验证 |
| **考纲编号一致性** | `syllabus_codes` 是否与考纲总览一致 | 对照 `[[中国化学奥林匹克基本要求-总览]]` |
| **图片引用** | `![[...]]` 路径是否正确（特别注意 mineru 路径前缀） | `Glob` 或 `Read` 图片本身 |
| **LaTeX 公式** | 渲染前先看源码是否有未闭合的 `$$` / `\(` | 文本扫描 |
| **重复与冲突** | 是否与现有同名知识点冲突？是否同一题被拆分了两次？ | `Glob` + `Grep` |
| **内容准确性**（抽样） | 公式、化学方程式、机理是否与原文一致 | 与源文件对照 |
| **跨系统连接**（v1.5.6 新增） | 本次产出是否建立了应有的跨系统双向链接？（见下方 §跨系统连接检查表） | `Grep` 验证下游引用 |

### 跨系统连接检查表（v1.5.6 新增）

> **目的**：防止"内容做对了，但跨系统导航链断裂"。每次构建/维护/备课任务 VERIFYING 阶段必须过一遍。

| 本次任务类型 | 必须检查的连接方向 | 验证方法 |
|:---|:---|:---|
| **提炼文件**（书籍/教学逻辑/真题） | 提炼页 → 对应 KP（wikilink 映射表）；若提炼涉及某专题 → 建议回链 [[04-专题与题型/专题-XXX]] | Grep 目标 KP/专题页中是否出现了该提炼文件 |
| **KP 填充/优化** | KP → 提炼源文件（frontmatter `sources`）；KP → 所属备课框架（如有） | Read KP frontmatter + Grep 提炼文件名 |
| **备课大纲** | 大纲 → 对应 [[04-专题与题型/专题-XXX]]（逐专题 wikilink）；大纲 → 提炼源文件 | 检查大纲 §一总览表是否有"对应专题页"列；Grep 提炼引用 |
| **备课框架修订** | 框架 §一总览表 "对应专题页" 列是否完整；§三 资源对接总表是否覆盖全部专题 | 逐行检查框架总览表 |
| **专题页创建/更新** | 专题页 → 提炼源文件（wikilink）；专题页 → 所属备课框架（反向引用） | Grep 对应框架中是否提及该专题页 |

> **发现断裂时的处理**：就地补齐 wikilink，并在 CLOSING 阶段工作日志中标注"补链 N 条"。不要推迟到"下次再说"。

### 修正模式

如果抽查发现问题：

| 问题规模 | 处理方式 |
|---------|--------|
| 单个文件单处错误 | 直接 `Edit` 修复 |
| 同类错误跨多个文件 | 启动"修正 Agent"，prompt 中明确"只修改 X 字段"以避免无关变动 |
| 系统性结构错误 | 暂停，回到 PLANNING 阶段重做计划，否则会污染整个批次 |

### REWORK 计数

如果同一任务需要多次修正（REWORK ≥ 2），应暂停执行，回到 ASSESSING 阶段重新评估任务类型和执行策略。

### VERIFYING 检查问句（自检）

- [ ] 我是否真的读了 3-5 个产出（不只是 1 个）？
- [ ] 我是否对照了源文件（至少 1 处）？
- [ ] 我是否检查了 wiki-link 不是红链？
- [ ] 我有没有跑过 Glob/Grep 来发现潜在重名或重复？
- [ ] **（v1.5.6）跨系统连接检查**：本次产出是否建立了应有的双向链接？（如果答不上来，先看上方 §跨系统连接检查表）
- [ ] 批量写入后，我是否抽查了 1-2 个文件的中文无 `�` 乱码？（见 [[经验沉淀]] 9-1）

## 2.5 CLOSING 阶段

> **目的**：让"今天做了什么"在未来仍可被检索；让"还没做完的"不会被遗忘。

### 脏模块审计

回顾 EXECUTING 阶段记录的脏模块清单，逐一处理或记录为 follow-up。

### 收工清单

快速检查：
- [ ] 所有产出文件是否已写入正确位置？
- [ ] frontmatter 必填字段是否完整？
- [ ] 跨系统链接是否已建立？
- [ ] 是否有未处理的 ⚠️ 标注？

### 任务卡生成 / 更新（必做）

> **目的**：把"还有什么没做"集中维护在一个任务卡系统里，避免 `工作日志` / `状态摘要` / `Agent战略方向` 多处重复写清单。

执行规则：
1. 本次新产生的未完成事项，优先在 `00-首页/活跃任务/` 新建任务卡
2. 本次已关闭的事项，在对应任务卡中改为 `status: completed` 并补 `completed`
3. `[[活跃任务]]` 只做聚合入口，不再手写整份短期任务面板
4. `[[工作日志]]` 只记录过程、依据、证据链接，不再复制整份短期任务面板
5. 若任务同时影响长期方向，再同步 `[[Agent战略方向]]` 的待决策项或优先级描述

### 出口清理：交付总结（给用户）

向用户的最终消息应包含：

1. **完成什么**：1-2 句概述 + 数量。
2. **产出在哪**：用 wikilink 给出 1-3 个代表性产出的路径。
3. **跳过/标注的项**：⚠️ 项的清单。
4. **建议下一步**：基于本次发现，建议下一个 Agent 任务做什么。

### CLOSING 检查问句（自检）

- [ ] 脏模块清单是否已处理或记录为 follow-up？
- [ ] 收工清单是否全部通过？
- [ ] 任务卡是否已更新？
- [ ] **任务卡已更新？`evidence` 已填？**（空 evidence = 任务未完成）
- [ ] **会话交接笔记已写？**（`00-首页/会话交接/YYYY-MM-DD-任务名.md`，15 行内）
- [ ] 给用户的总结是否点明了"完成 / 跳过 / 下一步"三件事？

# 三、通用红线

> 任何阶段中都不可越过的硬约束。

### 规则 1：相对路径
vault 内文件操作一律使用相对路径（如 `03-知识点/SN2反应.md`），**不**使用绝对路径或前导斜杠。

### 规则 2：不编造
源文件没有的内容禁止"补全"。要么留空、要么标注 `[待补充]` 或 `⚠️ 缺失`。

### 规则 3：未审核内容不进正式区
AI 输出默认进 `10-待审核区/` 或在文件 frontmatter 中标注 `status: 待审核`，不直接写入 `03-知识点/` 的"已填充"状态。

### 规则 4：保留原书结构
- 章节编号、题号、公式、化学方程式不改写；
- LaTeX 化学式必须用 `\mathrm{}`。

### 规则 5：操作可逆性优先
- 大批量删除 / 重命名前先在工作日志中记录"操作前快照"；
- 不要使用 `replace_all` 修改可能误伤的字符串（如普通词"反应"、"分析"）；
- **Edit 前先 `Grep` 验证 `old_string` 在目标文件中唯一**，避免批量替换时误伤（详见 [[经验沉淀]] 1-1）。

### 规则 6：与用户对齐
- 计划级 / 结构性变更（新建顶层目录、修改模板、批量改名）→ 必须先取得用户确认；
- 内容级填充（在已有骨架里写正文）→ 通常可直接执行，但产出后须在 CLOSING 阶段汇报。

### 规则 7：图片插入规范

> 详见 §六 配图规范。本条为简要引用：
> - 存放位置：同级 `assets/` 子文件夹
> - 链接格式：wiki-link `![[图名.ext]]`
> - KP 配图原则：仅 10-15% 的 KP 配图（几何/空间结构、反应机理、谱图典型、实验装置）
> - 配图是「备课/创建时的即时动作」，不是「维护期的批量补债」

### 规则 8：frontmatter 书写规范（v1.5 新增）

> 经验沉淀 1-1（全角冒号）和 1-6（status 引号包裹）的高频坑制度化。违反本规则会导致 Dataview 统计逃逸、Edit 工具失败或 frontmatter 解析异常。

- **字符串值不加引号**：`status: 已填充` ✅，`status: "已填充"` ❌（Dataview 可能将其识别为 `"已填充"` 而非 `已填充`，导致统计逃逸）

- **编辑 frontmatter 时先切英文输入法**，防止中文输入法混入全角符号
- 健康检查 Agent（Agent 9）定期扫描 `^status:\s*["']` 模式，报告引号包裹异常

# 五、PPT规范

> **触发条件**：本次任务涉及"生成 PPT 课件"或"将备课内容转为幻灯片"
> **工具位置**：`[[11-模板/pptx-toolkit/README]]`
> **内容规范**：`[[11-模板/Agent提示词-PPT生成]]`（六类幻灯片模板、认知台阶检查表、媒介分工提醒）

## 必须使用统一工具链

Agent 生成 PPT 时，**必须**使用 `11-模板/pptx-toolkit/` 下的全局工具，禁止直接调用 `pptxgenjs` 或 `html2pptx` 的底层 API：

| 文件 | 职责 | Agent 是否直接修改 |
|:---|:---|:---:|
| `11-模板/pptx-toolkit/ppt-engine.js` | PPT 生成引擎（封装 html2pptx + PptxGenJS + 公式插入） | ❌ 不修改 |
| `11-模板/pptx-toolkit/latex-render.js` | LaTeX → PNG 渲染引擎 | ❌ 不修改 |
| `11-模板/pptx-toolkit/模板-PPT生成.js` | 标准化生成脚本模板 | ✅ 复制后修改 |

## 公式必须走 LaTeX 渲染

**所有数学公式、化学方程式、离子表达式**，禁止以纯文本或图片硬编码方式放入 PPT，必须通过 `latex-render.js` 预渲染：

1. 在脚本中定义 `formulas` 注册表（key 以 `f_` 前缀 + 语义命名）
2. value 是纯 LaTeX 代码，**不要**包裹 `$` 或 `$$`
3. 在 `slides` 定义中通过 `formulas: [{ key, x, y, h }]` 指定位置和尺寸
4. `ppt-engine.js` 会自动批量渲染、缓存、插入

**为什么强制 LaTeX**：
- 保证公式风格统一（字体、颜色、尺寸）
- 缓存机制避免重复渲染，二次生成秒级完成
- 源文件可维护（修改 LaTeX 源码即可，不用重新画图）

## 书写规范速查

| 场景 | ✅ 正确 | ❌ 错误 |
|:---|:---|:---|
| 化学式 | `\\mathrm{[H_3O^+]}` | `[H_3O^+]`（会斜体） |
| 分式 | `\\frac{c}{K_a}` | `c/K_a`（排版差） |
| 文本下标 | `T = 298\\ \\mathrm{K}` | `温度 = 298K`（中文不支持） |
| 包裹符号 | `pV = nRT` | `$pV = nRT$`（不要 `$`） |

## 执行流程

```
1. 复制模板：cp "11-模板/pptx-toolkit/模板-PPT生成.js" "04-课件/PPT/{课程名}.js"
2. 填写内容：修改 OUTPUT_PATH、formulas、slides 三个区域
3. 运行生成：node "04-课件/PPT/{课程名}.js"
4. 输出验证：检查 formula-cache/ 是否正常生成、PPT 文件大小是否合理
```

## ⚠️ 已知限制（不踩坑）

| 问题 | 原因 | 解决 |
|:---|:---|:---|
| 公式图片溢出 | 公式太长或 `h` 设置过大 | 减小 `h` 或拆分公式 |
| `Cannot find module 'katex'` | pptx-workspace 未安装依赖 | `cd pptx-workspace && npm install pptxgenjs katex sharp playwright` |
| Playwright 浏览器报错 | Chromium 未下载 | `npx playwright install chromium` |
| 缓存不更新 | 修改公式但 key 没变 | 删除 `formula-cache/` 目录 |

> **SOP 版本更新时**：若 `pptx-toolkit/` 目录结构或 API 发生变化，需同步更新本规范，并迁移已有课件脚本。

# 六、配图规范

> 让"插图"成本降到最低，避免知识库里大量笔记"差一张图"。

## 6.1 图片插入规范

**存放位置**：
- 已配置 Obsidian 全局 `attachmentFolderPath: "./assets"`
- 任何笔记 Ctrl+V 粘贴 / 拖入图片 → 自动存到 **同级 `assets/` 子文件夹**
- 无需手动指定路径

**链接格式**：
- 一律用 wiki-link `![[图名.ext]]`，不用 `![](path)`
- 跨笔记复用图片用完整相对路径 `![[mineru/.../xxx.jpg]]`

**KP 配图原则（仅 10-15% 的 KP 配图）**：

| 类型 | 配图 | 例子 |
|---|:---:|---|
| 几何/空间结构 | ✅ | σ键、晶胞、椅式构象 |
| 反应机理 | ✅ | 箭头推动、过渡态 |
| 谱图典型 | ✅ | NMR/IR 特征峰 |
| 实验装置 | ✅ | 滴定、排水集气 |
| 概念/规则/公式 | ❌ | 焓、IUPAC命名、有效数字 |

配图后必须更新 frontmatter：`has_images: true` / `image_count: <N>` / `images_priority: high|medium`

## 6.2 frontmatter 字段语义（v1.3 知识点模板新增 `images_note` 字段）

| 字段组合 | 语义 | 用法 |
|:---|:---|:---|
| `images_note` 为空 | **暂未评估**(隐含债务) | 应在 KP 优化时(Agent 7)决策 |
| `has_images: true` + `images_note` 非空 | **已配图** | images_note 写"配了什么"(如 "图 5.5 一般形式 + 图 5.6 氮实例") |
| `has_images: false` + `images_priority: low` + `images_note` 非空 | **主动评估不需图**(已闭环) | images_note 写**评估理由**(如 "LaTeX 公式 + Mermaid 决策图已足够") |
| `has_images: false` + `images_priority: high` | **高优待补** | 强烈推荐补图;[[状态摘要]] §KP 图片覆盖率 自动统计 |

> 关键:`images_priority: low` + `images_note` 非空 = **主动评估**,不是"懒"。这样区别于"暂未评估"的隐含债务,[[状态摘要]] §KP 图片覆盖率 才不会把它误归为"高优待补"。
> 范例:[[Latimer图]] frontmatter(LaTeX + Mermaid 已足够,无需图像文件)。

## 6.3 配图触发条件（v1.4.4 新增）

> 控制配图成本的核心规则：**不要预先给所有 KP 批量配图**，只由「备课用到」或「创建/审阅时即时评估」触发。

| 场景 | 动作 |
|:---|:---|
| Agent 10 备课读取 KP，发现 `has_images: true` 但 `image_count: 0` | **触发 §6.4 检索 → 嵌入 → 更新 frontmatter** |
| Agent 10 备课读取 KP，发现 `images_note` 为空（未评估） | **快速决策**：这 KP 备课需要图吗？需要 → 标 high + 检索嵌入；不需要 → 标 low + 写评估理由 |
| Agent 1-6 创建/填充 KP 时，内容本身含几何/机理/谱图/装置 | **即时评估**：创建时就决定是否配图，一次性闭环 |
| `images_note` 已非空（无论 has_images 真假） | **跳过**，已评估闭环，不要回溯重评 |
| 用户未明确说「配图」的日常任务 | **默认不配**，纯文字 KP 是常态 |

**原则**：配图是「备课/创建时的即时动作」，不是「维护期的批量补债」。全部 KP 批量配图 ≈ 15-25 小时， ROI 极低；只有备课链路真实碰到的 KP 才值得花 3-5 分钟配图。

**优先级顺序**（备课/填充 KP 时按此顺序找图）：
1. **复用** mineru/ 和 06-外部资料导入/真题/images/ 已有图（见 [[图片索引]]）
2. **截图** 教材/文献 → Ctrl+V 粘贴
3. **手画** Excalidraw（机理、过渡态、能级图）
4. **代码** Mermaid 代码块（流程图、对比图、知识树）

**反哺工作流**(备课时课件→KP):
- 课件成形后挑 3-5 张关键图回填到对应 KP
- 在课件大纲 §六.5 列清单标记"已反哺/未反哺"

## 6.4 图片检索工作流（v1.4.1 新增）

> 当 Agent 备课 / 填 KP 需要找图时,按"三段式"流程定位,**不要凭空造图**。库内已有 7500+ 张 mineru OCR 图 + 真题原图,大概率主题已有图可用。

**一段:精标表速查(成本最低)**
1. 翻 [[图片索引]] §精标表 — 已沉淀过的主题直接命中
2. 命中 → 跳到第三段
3. 未命中 → 进入二段

**二段:文本检索路径(主用)**
```
Step 1: 多义/多语言关键词 Grep mineru/
       例: Grep "Markovnikov|马尔科夫尼科夫|马氏规则" → 命中 6 个 md
       例: Grep "钙钛矿|CaTiO|ABO3" → 命中 10 个 md

Step 2: 命中的 md 文件 → Grep '!\[' -n → 看图引用所在行号
       (mineru 图引用格式: ![](xxx_images/<hash>.jpg))

Step 3: Read 该 md 文件"图引用行 ± 10 行"范围
       图前后的题面/章节标题 = 图的天然语境
       例: 行 989 是"# 5.13 Frost 图" + 行 999 是首张图 → 高把握=图5.5

Step 4: 锁定 1-3 张候选(按上下文匹配度排序)
```

**三段:视觉验证(v1.4.2 强制)**

> ⚠️ **安全补丁(2026-05-17)**:取消"高把握跳过 Read"旧规则。实战发现 mineru OCR 会把二维码 / 版权页 / 印刷标记等无关元素也提取为图片,且**会插进章节正文之间**,仅凭"上下文写图 X.Y"判断会踩坑(详见 [[经验沉淀]] §2-6 + [[图片索引]] §⚠ OCR 噪声黑名单)。

- **强制规则**:对 mineru 来源的图,**首次接触某章节必须至少 Read 1 张采样验证**(确认 OCR 提取正常、上下文与实际图内容一致)
- **采样验证通过后**:同章节剩余的高把握图(上下文明确写"图 X.Y <主题>")可批采用
- **采样验证发现噪声**:在 [[图片索引]] §⚠ OCR 噪声黑名单 登记,跳过该图,继续后续候选
- **低把握**(上下文只说"如下图"或图分散):每张候选都要 Read
- **真题原图 / 自截图 / Excalidraw**:不受此规则约束,可直接采用
- 选定后嵌入:
  - 课件:`![[mineru/<路径>/<hash>.jpg]]`(跨目录长 wikilink,不复制文件)
  - KP:同上,并更新 frontmatter `has_images: true / image_count: N`

**沉淀(CLOSING 阶段同步做,关键步骤)**
本次找到的图 → 写入 [[图片索引]] §精标表:
| 主题 | 路径 | 关联 KP/题号 | 备注 |
下次同主题就能从一段直接命中,**避免每次重复检索**

## 6.5 图注笔记与 key_images 规范（v1.5 新增）

> **目的**：让图片从"插图"升级为"独立可搜索的知识对象"。一张图被 2 个以上 KP 引用，或在 2 道以上真题中出现时，建立图注笔记，实现"知识点 → 图 → 原始文件"的三角网络。

**何时创建图注笔记**：

| 条件 | 动作 |
|:---|:---|
| 一张图被 ≥2 个 KP 引用 | **必须**建图注笔记 |
| 一张图在 ≥2 道真题中出现 | **必须**建图注笔记 |
| 图是教材经典插图（如 Weller 图5.5） | **建议**建图注笔记 |
| 图仅在某一道真题中一次性使用 | 不必建，直接在真题文件中嵌入即可 |

**图注笔记创建规范**：

1. **位置**：`10-附件/图注/<主题>-<描述>.md`
2. **frontmatter**：
   ```yaml
   ---
   title: 图的自然语言标题
   type: 图片注释
   image_file: "[[完整相对路径.jpg]]"
   source: "来源描述"
   source_type: 教材插图 | 真题原图 | 自截图 | Excalidraw
   page_ref: "图号或题号"
   kp_links: ["[[关联KP1]]", "[[关联KP2]]"]
   tags: [图片, 主题标签]
   confidence: "🟢"  # 🟢已视觉验证 / 🟡仅上下文判断
   created: YYYY-MM-DD
   updated: YYYY-MM-DD
   ---
   ```
3. **正文结构**：
   - `# 标题` + `![[图片]]`
   - `## 图中关键信息`（3-5 条 bullet，提炼图中核心化学信息）
   - `## 教学要点`（表格：要点 + 说明）
   - `## 竞赛应用`（该图在初赛/决赛中的典型考法）
   - `## 关联真题`（wikilink 引用具体题目）
   - `## 复用场景`（该图适合在哪些专题/课件中复用）

4. **与 KP 的关联**：
   - 图注笔记创建后，立即更新对应 KP 的 frontmatter：
     ```yaml
     key_images: ["[[10-附件/图注/图注笔记文件名]]"]
     ```
   - `key_images` 上限 3 个，只放最核心的图。超过 3 个时，选"教学中最不可替代"的。

**图注笔记的使用链路**：

```
Agent 备课 / 答疑时
    ↓
读 KP frontmatter → 发现 key_images
    ↓
直接命中图注笔记 → 获取 image_file + 教学要点 + 关联真题
    ↓
嵌入课件 / 用于答疑
```

**与精标表的关系**：
- 精标表是"速查索引"（表格，一行一张图）
- 图注笔记是"深度卡片"（独立文件，可全文搜索）
- 两者互补：Agent 先用精标表定位图，再用图注笔记获取完整信息
- **CLOSING 阶段沉淀时**：新建图注笔记后，同步在精标表中添加「图注笔记」列的链接

**兜底:实在找不到**
1. Glob `mineru/*/` 看全目录,挑可能相关的(如教材某章)
2. 进该目录的 md → 搜近义词 / 上位词
3. 仍找不到 → 自己造图(优先级:Mermaid > 截图 > Excalidraw)

**示例(完整链路)**:
```
任务: [[钙钛矿结构]] KP 配晶胞图
一段: 翻 图片索引 — 未命中
二段: Grep "钙钛矿" mineru/ → 10 个 md

      → Read 该 md 行 11-30 看上下文 → 图位于"第 2 题钙钛矿题面"附近
三段: 上下文清晰,采用
嵌入: KP 添加 ![[mineru/04-其他试题/.../<hash>.jpg]]
决策: 钙钛矿结构图是否被 ≥2 个 KP 引用？是 → 创建图注笔记
创建: 10-附件/图注/钙钛矿结构-三种畸变示意.md
关联: 更新 [[钙钛矿]] KP frontmatter → key_images: ["[[10-附件/图注/钙钛矿结构-三种畸变示意]]"]
```

### 专题页聚合（备课任务专用）

> **触发条件**：本次任务类型为「备课 / 做大纲 / 整理讲义」且生成了新的备课大纲或新授课讲义
> **目的**：确保「课件产出」与「专题页知识库」同步，避免专题页长期为空导致复习课/习题课无法投影使用

**执行规则**：

| 专题页状态 | 动作 |
|:---|:---|
| `框架` | **必须聚合**：将本次备课中使用的核心结论、对比表格、解题套路、典型例题写入专题页对应 section，更新状态为 `已填充` |
| `已填充` | **增量更新**：追加新例题 / 修正错误 / 补充图片引用，**不覆盖**已有内容 |
| `已审核` | **只追加**：在典型例题串讲末尾追加新例题，frontmatter 中更新 `updated` |

**聚合内容清单**（按优先级）：
1. **核心结论汇总**（若为空，填充 1-3 句话 + 最高频决策路径）
2. **对比表格**（若为空，填充至少 3 行，含触发条件列）
3. **解题套路 / 决策流程**（若为空，填充至少 2 步，每步链接具体 KP）
4. **典型例题串讲**（追加至少 1 道完整例题：题目+分析+解答+反思）
5. **相关课件与讲义**（自动添加反向链接，Agent 10 已执行）
6. **教材原图引用**（自动添加图片引用，Agent 10 已执行）

**最小可用标准**（专题页从 `框架` → `已填充` 的门槛）：
- 核心结论汇总 ≠ 空
- 对比表格 ≥ 3 行
- 解题套路 ≥ 2 步
- 典型例题串讲 ≥ 1 道完整例题

**降级策略**：
- 若本次备课未覆盖某 section（如机理类专题未用到反应机理拆解），则跳过该 section，不强行填充
- 若专题页已有内容但与本备课视角冲突（如专题页写的是基础班视角，本次是提高班视角），在例题串讲中**追加**而非覆盖，用标注区分班型

**检查问句**：
- [ ] 本次备课对应的专题页文件路径是什么？状态是什么？
- [ ] 若状态为 `框架`，是否已完成最小可用填充（结论+表格+套路+例题）？
- [ ] 专题页中的 HTML 锚点（{#core-conclusions} 等）是否正确存在？
- [ ] 专题页 frontmatter 中的 `status` 和 `updated` 是否已更新？

> **为什么这是 Session 终结而非 EXECUTING 阶段**：专题页聚合是"归档动作"，应在课件内容确定后执行，避免备课过程中反复修改专题页导致内容漂移。

## 7.3 同日Follow-up去重

> **目的**：防止同一天工作日志里多个 H2 段落各自复制一份"下次会话可推进"，后段已完成、前段仍写未完成，造成状态漂移。

执行规则：
1. 若同一日文件内存在多个 `#### 🔥 高优先级 — ...` 块，**最新块为权威块**；
2. 当后续 H2 完成了前文 Follow-up 中的任务，需回填前文对应条目为完成态，至少更新最近一次出现的旧块；
3. 新增 Follow-up 时优先新建任务卡；若确需在日志里保留，也优先使用任务语法：
   - 未完成：`- [ ] **任务名**：说明`
   - 已完成：`- [x] ~~**任务名**~~ → YYYY-MM-DD 已完成`
4. 若因历史原因保留编号列表，Session 终结前必须人工检查是否存在"前文未完成、后文已完成"的冲突。

## 7.4 历史高优待办清理

> **目的**：防止历史工作日志中的「实际已完成但未打勾」任务持续污染 Dataview 高优 Follow-up 看板。

**触发条件**：
- 本次会话涉及 **维护 / 审计 / 批量修正 / 跨日 follow-up 处理**
- 或用户在会话末尾明确要求「收尾 / 记录待办 / 准备结束对话」

**执行范围**（限制边界，防止无限回溯）：
1. 只查 `00-首页/工作日志/` 下**最近 7 天**的文件；
2. 只查处于 `#### 🔥` 标题下的 `- [ ]` 任务；
3. 只改**明确 100% 已完成**的任务（如 KP 已填充、脚本已执行、文件已迁移）；
4. **不修改**跨会话持续推进的任务描述（如「路线 B2'：决赛 11 骨架填充」改为 100% 前，不在旧日志中回填）。

**修改格式**：
```markdown
- [x] ~~**任务标题**：原描述~~ → YYYY-MM-DD **已完成**（简要说明）
```

**示例**：
```markdown
- [x] ~~**真题 knowledge_points 覆盖率提升**：33.4% → 60%+~~ → 2026-05-14 **已完成**（全量脚本，覆盖率 ~86%）
```

> ⚠️ **不要删除条目**（保留 `~~删除线~~` 和完成时间），以维持历史可追溯性。
