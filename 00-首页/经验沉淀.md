## 一、Frontmatter / YAML 类

### 1-1 全角冒号导致 Edit 失败 🔥
- **现象**：用 Edit 工具修改 frontmatter 时，old_string 与文件实际内容看似一致，但工具报「old_string not found」。
- **原因**：YAML 中的冒号必须是半角 `:`（U+003A）；如果用全角 `：`（U+FF1A），表面看相同，但字节不同。中文输入法下打字极易混入全角。
- **规避方案**：
  1. 写 frontmatter 时先切英文输入法
  2. Edit 失败时第一反应：`Grep -P ":\s" file` 检查是否混入全角
  3. 用 Read 工具核对原文（cat -n 输出会保留原字符）
- **案例**：2026-05-04 化学原理批次填充时多次撞坑

### 1-2 frontmatter 未闭合导致整文件被识别为 YAML
- **现象**：Dataview 查询不到该笔记，Obsidian 大纲面板异常。
- **原因**：frontmatter 必须以独立一行 `---` 开头并以独立一行 `---` 结尾。如果第二个 `---` 前有空行/缩进/或被误删，整个 .md 都被吃进 YAML 解析器。
- **规避方案**：
  1. 写 frontmatter 模板时复制 [[模板-知识点]] 完整开头
  2. Edit 时永远不动 `---` 行，只动中间的 key: value
- **案例**：2026-04 模板初版踩过

### 1-3 数组字段格式不一致破坏 Dataview 聚合
- **现象**：Dataview `GROUP BY subject` 时同一类目被拆成多个组（"无机化学" 与 "无机和结构化学" 各自一行）。
- **原因**：subject/tags 等枚举字段没有规范化。同一概念有多种写法（"无机" / "无机化学" / "无机和结构化学"）。
- **规避方案**：
  1. 在 [[模板-知识点]] 中明确列出 `subject` 的可选值清单
  2. 健康检查 Agent（Agent 9）定期审计 frontmatter 字段一致性
- **案例**：2026-05-06 配置 Dataview 库存表时发现

### 1-4 status 字段三态约定
- **约定**：仅使用「骨架 / 初稿 / 已填充」三态，不允许出现「完成」「填充中」「待审」等同义词
- **原因**：状态摘要的 Dataview 查询硬编码了这三态；新增态会导致统计不全
- **如要扩展**：必须先改 [[状态摘要]] 的 Dataview，再批量改 KP

### 1-5 "暂未评估" vs "主动评估不需图" 的语义区分
- **现象**：v1.2 模板的 `has_images: false / images_priority: low` 字段组合无法区分"懒得评估"与"评估过不需要"两种情况,导致 [[状态摘要]] §KP 图片覆盖率 把已评估的 KP 误统计为"待补"
- **触发**:实战测试 [[Latimer图]] 时——LaTeX 公式已表达完整,加 Mermaid 决策图作教学辅助即可,**主动判断不需要图像文件**,但 frontmatter 无法表达这个决策
- **规避方案(v1.3 模板引入)**:
  - 新增 `images_note: ""` 字段记录评估理由
  - 三态语义:
    - `images_note` 为空 → **暂未评估**(隐含债务)
    - `has_images: true` + note 非空 → **已配图**(note 写配了什么)
    - `has_images: false` + `images_priority: low` + note 非空 → **主动评估不需图**(已闭环,无债务)
  - 完整字段语义表见 [[Agent标准作业流程]] v1.4.3 规则 7
- **案例**:2026-05-17 [[Latimer图]] 标准用法:
  ```yaml
  has_images: false
  image_count: 0
  images_priority: low
  images_note: "LaTeX 公式 + Mermaid 决策流程图已足够;原书 §5.12 不含图像截图,无需补图。"
  ```

### 1-6 status 字段引号包裹导致统计逃逸 🔥
- **现象**：状态摘要声称"698/698 已填充"，但 PowerShell 扫描仅 674 个 `status: 已填充`（无引号格式）。两者相差 24 个文件。
- **原因**：历史批量操作中 28 个 KP 的 status 被写成 `""已填充""`（双引号包裹内容），1 个写成 `"已填充"`（单引号包裹）。PowerShell 正则 `^status:\s*已填充` 无法匹配 `"` 字符，Dataview YAML 解析器也可能将其识别为 `"已填充"`（含引号作为内容）。
- **规避方案**：
  1. **Agent 9 健康检查增加规则**：扫描 `^status:\s*[""'']` 模式（status 值以引号开头），报告异常
  2. **批量创建/迁移脚本强制规范**：写入 frontmatter 时，字符串值不加引号（除非值内含 YAML 特殊字符如冒号、井号）
  3. **状态摘要核对**：PowerShell 统计后手动校验 `已填充 + 骨架 + 初稿 + README = 总数`
- **案例**：2026-06-01 全库审计发现 29 个 status 引号包裹 + 8 个旧值 `框架`，批量修复后统计对齐（703 已填充 / 11 骨架 / 0 初稿）

### 1-7 `KP-` 前缀旧文件与正式 KP 重复 🔥
- **现象**：知识库中同时存在 `KP-晶体场理论.md`（初稿）和 `化学原理/晶体场理论.md`（已填充），内容重复但前者 status 为初稿、内容更薄。
- **原因**：早期批量创建时使用了 `KP-` 前缀作为临时标识，后续正式 KP 建立后未清理旧文件。
- **规避方案**：
  1. **Agent 9 健康检查增加规则**：扫描文件名以 `KP-` 开头的文件，检查是否存在同名（去掉 `KP-` 前缀）的正式 KP；若存在且正式 KP 状态 ≥ 初稿 → 标记为重复待删除
  2. **新建 KP 时不再使用 `KP-` 前缀**：直接用概念名作为文件名
  3. **审查时增加 "前缀去重" 步骤**：合并/弃用 Agent（Agent 8）处理 `KP-` 文件时，优先匹配去掉前缀后的文件名
- **案例**：2026-06-01 删除 4 个重复初稿（`KP-晶体场理论`、`KP-配合物`、`KP-异构体`、`KP-杂化轨道`），均为已有正式 KP 的冗余副本

### 1-8 学生讲义缺乏图片追踪 frontmatter 字段 🔥
- **现象**：学生讲义（`04-课件/学生讲义/`）中没有任何图片相关的 frontmatter 字段（如 `has_images` / `image_count`），无法通过 Dataview 自动统计配图覆盖情况，每次统计不得不手动 Grep `![[media/` 模式。
- **原因**：讲义模板未定义图片追踪字段。KP 有 `has_images`/`image_count`/`images_priority` 三字段体系，但讲义从未纳入此体系。
- **规避方案**：
  1. **短期**：统计时用 `Grep "!\[\[media/" 04-课件/学生讲义/*.md` 做模式匹配
  2. **长期**：在讲义模板中加入 `has_images` / `image_count` 字段，与 README 配图统计表形成双源校验
- **案例**：2026-06-24 配图战役最终统计时，54 份讲义的配图数全部通过 Grep 手工汇总，无法用 Dataview 自动化

## 二、Wikilink / 路径类

### 2-1 wikilink 不带 .md 后缀 🔥
- **现象**：写 `[[文件名]]` 时 Obsidian 显示为绿色（未解析）链接，跳转失败。
- **原因**：Obsidian wikilink 默认对应文件名，不需要 .md 扩展名。
- **规避方案**：
  - ✅ `[[Lawesson试剂]]` 或 `[[03-知识点/有机化学/Lawesson试剂]]`
  - ❌ `[[Lawesson试剂]]`
- **例外**：Markdown 标准链接 `[文本](路径.md)` 反而需要带 .md
- **案例**：高频踩坑，几乎每个新 Agent 第一次都会犯

### 2-2 wikilink 中目录路径斜杠方向
- **现象**：Windows 下用反斜杠 \ 写路径，跨平台同步后变成普通文本
- **原因**：Obsidian wikilink 解析器只认正斜杠 /；反斜杠在 Linux/macOS 下是普通字符，跨平台同步后路径解析失败
- **规避方案**：wikilink 内部一律用正斜杠 `/`（Obsidian 跨平台规范）
- **案例**：Read 工具用 `\`，Edit 写 wikilink 用 `/`，注意区分

### 2-3 重命名后留下断链
- **现象**：把 KP 从「过氧化氢」改名为「过氧化氢及过氧化物」后，所有指向它的 wikilink 全部断掉
- **原因**：手动改文件名时 Obsidian 不知道这是重命名（认为是删除+新建），无法触发反向引用的自动更新机制
- **规避方案**：
  1. 用 Obsidian 内置「重命名」（自动更新所有引用），不要直接改文件名
  2. AI 做重命名前必须先 `Grep "过氧化氢"` 列清单，逐一改完后再改文件名
- **案例**：[[Agent提示词-知识库维护]] 的 Agent 8（合并/弃用）必须遵守

### 2-4 别名（aliases）未使用导致同名分裂
- **现象**：「Lawesson 试剂」和「Lawesson试剂」被视作两个不同 wikilink
- **原因**：Obsidian 默认按字面比对 wikilink 文本，半角空格差异等于不同标识；aliases 是显式声明的同义词集合
- **规避方案**：在 frontmatter 中加 `aliases: [Lawesson 试剂, 拉森试剂]`，让多种写法都指向同一笔记
- **案例**：化学命名易踩，建议核心 KP 都加 aliases

### 2-5 图片插入路径未统一导致每次手敲路径 / 链接易断
- **现象**：早期笔记图片插入需要手动指定文件夹路径（Obsidian 默认让用户选目录），导致：
  - 图片散落在仓库各处（`mineru/.../images/`、`06-外部资料导入/.../images/`、根目录等）
  - 每个笔记插图都要重新敲一遍路径，备课/填 KP 时摩擦大
  - 笔记重命名/移动后图片链接易断
- **原因**：`.obsidian/app.json` 未配 `attachmentFolderPath`，Obsidian 使用"提示用户选择目录"的默认行为
- **规避方案**：
  1. 全局配置（2026-05-17 起已生效）：
     ```json
     "attachmentFolderPath": "./assets",
     "newLinkFormat": "shortest",
     "useMarkdownLinks": false
     ```
  2. Ctrl+V 粘贴图片 → 自动存到笔记同级 `assets/`，自动生成 `![[图名.ext]]`
  3. 跨笔记复用图片用完整相对路径 `![[mineru/.../xxx.jpg]]`
  4. 详细规范见 [[Agent标准作业流程]] 规则 7
- **案例**：2026-05-17 知识库整改前 297 个 KP 几乎无配图，根本原因之一是插图摩擦过大。整改后通过 Ctrl+V 即可完成"截图→入库→链接"三步合一

### 2-6 mineru OCR 噪声：二维码 / 版权页混进章节内 🔥
- **现象**：mineru 把 PDF 的二维码、版权页、印刷标记等无关元素也提取为图片，且**位置可能落在正文章节中**，导致：
  - 上下文文字写"图 X.Y"，但实际紧邻的图是无关元素
  - Agent 若按 SOP §7.X "高把握跳过验证"规则直接采用，会把二维码嵌进 KP
- **原因**：mineru OCR 按 PDF 页面坐标抽取图片，无法区分"正文图"与"装订噪声"；图片顺序可能与原书图编号不一致
- **规避方案**：
  1. **SOP §7.X 修订**：取消"高把握跳过 Read"规则，**任何 mineru 章节首次接触必须至少 Read 1 张采样验证**；同一章节内验证后可信任
  2. 已识别的"陷阱图"登记到 [[图片索引]] §OCR 噪声黑名单（Agent 命中候选时先查黑名单）
  3. 若候选图视觉验证发现是噪声 → 看下一张图引用（章节内顺序常错位 1-2 张）
- **案例**：2026-05-17 实战测试 [[Frost图]] 配图时，源 md 行 1015 上下文写"图5.6"实为**二维码**，行 1018 才是真正的"图5.6 氮的 Frost 图"。已入黑名单防再次踩坑

### 2-7 OCR 导入笔记的 Markdown 图片语法与 HTML 断链批量修复 🔥
- **现象**：大量 OCR 工具（如 mineru）导入的笔记中，图片使用标准 Markdown 语法 `![](folder/xxx.jpg)` 而非 Obsidian WikiLink `![[folder/xxx.jpg]]`；HTML 表格内残留 `<img src="images/xxx.jpg"/>` 指向不存在的 `images/` 文件夹；配套 `_images` 文件夹名含中文括号 `（）`，在跨平台同步和脚本处理中易触发编码/解析错误
- **原因**：
  1. OCR 工具默认输出标准 Markdown 图片语法，不针对 Obsidian WikiLink 生态优化
  2. 工具将图片抽入 `_images` 子文件夹，但 HTML 表格内的 `<img>` 标签仍保留源 PDF 的相对路径 `images/`
  3. 中文括号 `（）` 在 URL 编码、PowerShell 正则、部分文件系统 API 中会被转义或解析异常，导致批量脚本匹配失败
  4. 文件夹重命名后，若脚本从文件系统读取"当前文件夹名"去替换 md 内容，会与 md 中嵌入的"旧文件夹名"失配（因为 md 内容尚未更新）
- **规避方案**：
  1. **Markdown → WikiLink 批量转换**：扫描 `!\[\]\(([^)]+_images)/([^)]+)\)`，替换为 `![[$1/$2]]`，使图片链接纳入 Obsidian 原生管理
  2. **HTML `<img>` 断链修复**：将 `<img src="images/` 替换为 `<img src="实际文件夹名_images/`，注意 HTML 表格内必须保留 HTML `<img>` 格式（WikiLink 在 HTML 块中不被解析）
  3. **清理中文括号**：重命名 `_images（N）` → `_images_N`，并在 md 文件中同步更新所有引用路径
  4. **提取旧名必须从文件内容而非文件系统**：文件夹已重命名后，文件系统返回的是新名；必须从 md 内容中 regex 提取旧的 `_images` 文件夹名作为替换锚点，否则替换率为 0
  5. **写入后双归零验证**：扫描 `!\[\]\(` 确认 Markdown 语法归零；扫描 `<img src="images/` 确认 HTML 断链归零
- **案例**：2026-06-05 全库扫描发现 167 个文件、14,779 处 Markdown 图片语法 + 687 处 HTML 断链 + 42 个含中文括号的 `_images` 文件夹；分三批次修复（语法转换 → 文件夹重命名 → HTML 断链），最终验证双归零

### 2-8 配图命名规范不统一（hash vs 描述性命名）
- **现象**：media/ 目录下同时存在三类命名方式：
  - **描述性命名**：`13-4-bragg-reflection.jpg`（可读、可搜索）
  - **SHA256 截断+描述混合**：`ef013960-fcc-unit-cell.jpg`（前半段哈希+后半段描述，不一致）
  - **完整 SHA256 哈希**：`3ec6f85bbab0ee45a207978a1437f5e6aff9e27d662bf16f38f982170797b526.jpg`（完全不可读）
- **原因**：不同批次的配图 Agent 采用了不同的命名策略——从 mineru 拷贝时默认保留哈希名，后续手工重命名时又改为描述性命名；两套策略的产物长期共存。
- **规避方案**：
  1. 统一采用 `{chapter}-{figure}-{简短英文描述}.jpg` 格式（如 `13-4-bragg-reflection.jpg`）
  2. 存量哈希名图片在下次使用到时可逐步重命名（同时更新所有引用该图片的讲义）
  3. 新拷贝图片一律按描述性命名入库，禁止直接引用 mineru 哈希名路径
- **案例**：2026-06-24 配图战役中，5 张新拷贝图片全部采用描述性命名，但 media/ 仍存有 ~50 张哈希命名图片（含完整哈希与截断混合两类），需逐步治理

## 三、LaTeX / 化学式类

### 3-1 化学式必须用 \mathrm{} 🔥
- **现象**：直接写 `$H_2O$` 渲染成 *H₂O*（H 是斜体，违反 IUPAC 规范）
- **原因**：LaTeX 默认数学模式下字母按斜体变量渲染（`H` → 物理量），但化学元素符号是直立字体（IUPAC 规范）；`\mathrm{}` 强制 roman 直立体
- **规避方案**：
  - ✅ `$\mathrm{H_2O}$` 或 `$\ce{H2O}$`（mhchem 扩展）
  - ❌ `$H_2O$`
- **复杂情况**：含正/负电荷（`$\mathrm{H_3O^+}$`）、相态（`$\mathrm{(aq)}$`）、同位素（`$^{14}\mathrm{C}$`）都要套 mathrm
- **案例**：化学原理与无机批次填充每次都要复检

### 3-2 mhchem 包未启用时 \ce 失效
- **现象**：`$\ce{H2O}$` 渲染成原始 `\ce{H2O}` 文本
- **原因**：`\ce` 是 mhchem 扩展包提供的化学方程式专用命令，若 MathJax 未加载此扩展则 `\ce{}` 被识别为未知命令而原样输出
- **规避方案**：本库已通过 Obsidian 设置启用 mhchem。如发现失效，先检查 `编辑器 → MathJax 配置`
- **案例**：迁移 Obsidian 版本/配置时易踩

### 3-3 反应箭头要用专用命令
- **现象**：直接打 Unicode 箭头（→ ⇌ ↔）在 `$...$` 内不渲染或渲染不正确
- **原因**：LaTeX 数学模式下 Unicode 字符不被解析为符号，会被当成普通文本（甚至完全消失）；化学/物理反应箭头需要 LaTeX 命令配合上下注才能正确编译
- **规避方案**：
  - 单箭头 `→`：`\rightarrow` 或 `\to`
  - 双箭头平衡 `⇌`：`\rightleftharpoons`
  - 大箭头加注 `→[催化剂]`：`\xrightarrow{\text{催化剂}}`
- **避免**：直接打 Unicode 箭头（在 `$...$` 内不渲染）
- **案例**：化学方程式渲染（如 催化剂 等机理类 KP 的反应式）

### 3-4 LaTeX 块内中文注释会断行
- **现象**：`$E = mc^2$ 这是质能方程` 内若中英文混排过多会换行错乱
- **原因**：行内公式 `$...$` 会被 MathJax 视作不可换行的整体节点，长公式后接中文文本时浏览器换行算法会把整行拽到下一行，造成视觉错乱
- **规避方案**：行内公式短即可，复杂公式用块级 `$$...$$` 单独成块
- **案例**：爱因斯坦质能方程 类笔记多次重排

### 3-5 反应机理图的箭头推动符号
- **现象**：箭头推动机理图无法用纯 LaTeX 表达，必须靠图
- **原因**：箭头推动是机理学的'拓扑信息'（电子从哪到哪），二维几何关系无法用一维文本流表达，必须靠图（ChemDraw 截图）
- **约定**：箭头推动笔记中
  - 双钩箭头（电子对流动）：用 `\curvearrowright` 或文字描述
  - 单钩箭头（单电子）：用文字标注「单电子转移」
  - 实际渲染靠图（用 ChemDraw/MarvinSketch 截图嵌入），不强求 LaTeX 渲染
- **案例**：[[Arrow Pushing in Inorganic Chemistry-总索引]] 系列

## 四、Edit 工具操作类

### 4-1 old_string 必须唯一 🔥
- **现象**：Edit 报「old_string is not unique」
- **原因**：Edit 工具语义是'精确定位 + 替换'，多个匹配位置时无法判定哪一个是用户意图，工具直接 abort 而非随机或全选
- **规避方案**：
  1. 扩大 context（多带前后 1-2 行的上下文，特别是行首/行尾的特殊字符）
  2. 慎用 `replace_all: true`（会改全文，难以预测副作用）
  3. 终极方案：先 Read 看清行号，再用更长片段做唯一锁定
- **案例**：填充模板时的 14 个三级标题（## 一/二/三...）极易冲突

### 4-2 编辑前必须先 Read
- **现象**：未 Read 直接 Edit 工具会报错拒绝执行
- **原因**：Read 操作建立的'已知文件状态'是 Edit 的输入前置；未 Read 直接 Edit 等于基于猜测内容做修改，工具内置校验拦截以避免错配
- **强制规则**：Edit 工具内置校验——未 Read 过的文件不能 Edit
- **理由**：避免基于过期/猜测内容做修改
- **规避方案**：每个新会话第一次改某文件前，先 Read 一次（即使是上轮已读过的文件）

### 4-3 Tab vs Spaces 不一致
- **现象**：复制粘贴他人内容（特别是从浏览器/PDF）时混入 Tab，但本库一律用 4 spaces
- **原因**：Tab 在不同编辑器/渲染器下宽度不一致（4/8/可变），Markdown 列表层级又依赖空格数判定，Tab 混入会破坏列表/缩进解析
- **规避方案**：粘贴大段内容后用 `Grep -P "\t"` 检查 Tab；批量替换为空格
- **案例**：粘贴 PDF 公式块时易踩

### 4-4 Edit 后行号变化导致后续 Edit 失败
- **现象**：连续多个 Edit 操作，第二个 Edit 时引用了第一个 Edit 改变前的行号
- **原因**：每次 Edit 都会改变后续行号；硬编码行号的二次 Edit 等于基于过时坐标定位，行号差几行就完全错位
- **规避方案**：
  1. 同一文件的多个 Edit 操作必须串行，每次 Edit 后行号变化
  2. 不要在 prompt 里硬编码行号，靠 old_string 唯一性定位
- **案例**：批量改 frontmatter 时的典型坑

### 4-5 Write 会覆盖整个文件
- **现象**：本想追加内容，结果用了 Write 把全文覆盖
- **原因**：Write 工具语义是'整体重写'（不是'追加'也不是'片段替换'）；混淆工具语义会丢失文件原内容
- **规避方案**：
  - 修改现有文件 → **必用 Edit**
  - 创建新文件 / 完全重写 → 才用 Write
  - 追加到末尾 → Edit + 找文件最后一行做 anchor

### 4-6 PowerShell 双引号字符串内反引号 escape 陷阱 🔥
- **现象**：用 PowerShell 批量替换字符串时，写入文件后发现：
  - `` `aliases` `` 被切成 `liases`（首反引号 + a → BEL/U+0007，被吞）
  - `` `H_2O` `` 反引号集体丢失变 `H_2O`
  - `` `\ce{}` `` 反引号丢失变 `\ce{}`
  - 文件随机位置混入 BEL（U+0007）、`` `n `` 变换行符等不可见破坏
- **原因**：PowerShell 双引号字符串 `"..."` 内反引号 `` ` `` 是 escape 字符（不是字面量）：

### 4-7 Obsidian 预览切换报错未必是当前笔记/Dataview 本身，可能是日历插件干扰 🔥
- **现象**：某个 Markdown 笔记可以正常打开，但一执行 `markdown:toggle-preview` 就报错，导致误以为是当前笔记里的 Dataview / DataviewJS 查询写坏了。
- **本次报错**：
  ```text
  TypeError: this.currentMode.getEphemeralState is not a function
  ```
- **原因**：这类错误不一定来自当前页面内容，也可能来自插件改写了 Markdown 视图 / 叶子状态切换逻辑。本次实测中，触发源是社区插件 `dust-calendar v1.4.0`，而不是 `[[02-数据库/数据库-真题]]` 的 Dataview 查询。
- **规避方案**：
  1. 先清空 `dev:errors`，区分“历史错误”与“当前动作新触发错误”
  2. 分离动作验证：
     - 只 `open` 页面，如果无错，说明不是“页面一加载就炸”
     - 再单独执行 `markdown:toggle-preview`，确认是否是“预览切换链路”触发
  3. 优先排查会接管视图 / 面板 / 日历 / 工作区状态的插件，而不是先怀疑 Dataview
  4. 用“最小可逆实验”定位：
     - 保持 Dataview 不动
     - 先临时禁用最可疑插件
     - 复测预览切换是否恢复
  5. 若禁用某插件后报错立刻消失，就先记录为兼容性问题，再决定是否长期停用
- **案例**：2026-06-12 运行态排查 `[[02-数据库/数据库-真题]]` 时，执行 `markdown:toggle-preview` 抛出 `this.currentMode.getEphemeralState is not a function`；临时禁用 `dust-calendar` 后，页面成功进入 `preview`，说明根因是插件链路而非题库页查询本身。
  - `` `a `` → BEL（U+0007，alarm bell）
  - `` `n `` → LF 换行
  - `` `t `` → Tab
  - `` `H `` / `` `e `` 等无效 escape → 反引号被吃，字母保留
  - **跨行** here-string `@"..."@` 同此规则
- **规避方案**：
  1. **替换字典的 value 用单引号字符串**：`@{ key = '反引号 \`xxx\` 安全' }`（单引号内 `` ` `` 是字面量）
  2. **必须用双引号时，反引号写两个**：`"反引号 \`\`xxx\`\` 才是字面量"`（PowerShell 中 `` `` `` 是 escape 反引号字面量）
  3. **批量替换写入后立即字节级扫描**：`[regex]::Matches($content, [char]0x07).Count` 应为 0；同时扫 U+000B / U+0085 / U+2028 / U+2029 / U+200B
  4. **小样本测试 → 抽查 → 再批量**：先替换 2-3 条，Read 工具核对反引号无丢失，再扩到 27 条
  5. **首选 Edit 工具**：单条精确替换永远比 PowerShell 批量安全；只有 ≥10 条同结构改动且 anchor 不冲突时才考虑 PowerShell
- **诊断**：批量替换后做四步快速诊断：
  ```powershell
  $c = [IO.File]::ReadAllText($path, [Text.UTF8Encoding]::new($false))
  # 检查 BEL（最高频）
  ([regex]::Matches($c, [char]0x07)).Count
  # 检查反引号配对
  ($c -split "`n" | Where-Object { (($_.ToCharArray() | Where { $_ -eq '`' }).Count % 2) -eq 1 }).Count
  # 检查关键词完整（如 aliases 没被切）
  ([regex]::Matches($c, '\bliases\b')).Count   # 应为 0
  # 检查不可见字符总数
  ([regex]::Matches($c, '[\x{0007}\x{000B}\x{0085}\x{2028}\x{2029}\x{200B}]')).Count
  ```
- **字符串替换 API 选用**：批量替换写脚本时还要在 `.Replace()` 与 `-replace` 之间做选：
  - `.Replace(old, new)` 是 **字面量替换**，源串中的 `()`、`（）`、`,`、`.` 等正则元字符都按字面处理；搭单引号字符串最安全
  - `-replace 'pattern', 'new'` 是 **正则替换**，源串里若含元字符必须 `[regex]::Escape($pattern)` 预处理，否则匹配范围会膨胀
  - **判据**：替换源是「字面路径 / 字面 wikilink / 字面字段名」→ `.Replace()`；替换源是「带通配的模式」→ `-replace`
- **案例**：2026-05-10 [[经验沉淀]] 27 条原因字段批量补齐时踩，27 条中 5 条反引号丢失 + 1 条 `aliases` → `liases`（5 处BEL混入）；当时 Edit 工具因 old_string 含 BEL 无法匹配，最终用 `[char]0x07` 显式锚点修复。完整复盘见 [[00-首页/工作日志/2026-05-10]] 的「📅 2026-05-10 经验沉淀『四要素强治理』全量补齐」H2 §关键决策 C；同日 mineru/ 901 处批量断链修复（[[工作日志/2026-05-10]] L1605）验证 `.Replace()` + 单引号字典是含元字符场景的最稳工具组合

### 4-7 PowerShell `[IO.File]::ReadAllText/WriteAllText` 绕过 Edit "Read-first" 限制
- **现象**：批量跨文件 wikilink 替换（如把 `mineru/.../images/xxx.jpg` 全部改成新路径）涉及 10+ 文件，Edit 工具要求每个文件先 Read 再 Edit，同一消息内并行处理被节流；分轮串行又会让 Agent 上下文膨胀
- **原因**：Edit 工具有「Read-first」语义约束（避免基于过期内容修改）；Read 跑 10+ 文件 = 10+ 轮工具调用 = 上下文窗口快速耗尽；同 PR 内大规模一致替换需要一次性原子操作
- **规避方案**：用 `.NET` 直接读写（绕过 Edit 校验），且强制 UTF-8 不带 BOM：
  ```powershell
  $enc = [Text.UTF8Encoding]::new($false)    # $false = 不带 BOM
  Get-ChildItem -Recurse -Filter *.md | ForEach-Object {
    $c = [IO.File]::ReadAllText($_.FullName, $enc)
    $new = $c.Replace('mineru/旧路径', 'mineru/新路径')  # 字面量替换（见 4-6）
    if ($new -ne $c) { [IO.File]::WriteAllText($_.FullName, $new, $enc) }
  }
  ```
- **适用边界**：
  1. **适用**：≥10 文件同结构替换 + 替换源为字面量 + 无需 Edit 校验保护（即「我已确认要批改」）
  2. **不适用**：单文件精细修改（用 Edit）/ 替换源含正则元字符（先 `.Replace()` 试，否则换 `-replace + [regex]::Escape()`）/ 文件含 BOM 但其他工具又依赖 BOM（先 grep 一遍 BOM 决定编码）
- **写入后必验**：① `Before -ne After` 比 `Match.Count` 准（包含 markdown 形式之外的纯文本路径）；② Read 抽样 1-2 个文件核对反引号 / 中文 / unicode 字符无破坏；③ 跑一次 Dataview 看断链情况
- **案例**：2026-05-10 mineru/ 子目录整理引发 5 类 901 处 wikilink 断链，用 `.Replace()` + 单引号字典 + 本规范一次性修复 67 个文件（[[工作日志/2026-05-10]] L1604-1606）；同模式已用于 Phase C 第二批 [[β-硅基效应]] 重命名后的 12 处反向引用同步

### 4-8 PowerShell `-replace` 动态正则的参数解析陷阱 🔥
- **现象**：用 PowerShell `-replace` 做批量字符串替换时，若右侧参数（替换字符串或正则模式）是动态构造的变量，且其中包含空格、逗号、括号等字符，报错 "The -replace operator allows only two elements to follow it, not 4" 或 "The regular expression pattern ... is not valid"。82 个文件的批量替换因此连续失败
- **原因**：`-replace 'pattern', 'new'` 的右侧两个参数通过 PowerShell **命令解析器**处理；当变量 `$pattern` 或 `$new` 展开后包含空格/逗号/括号时，解析器将其误认为额外的参数分隔符，导致参数数量或正则语法错误。这与 4-6 的"字符串内反引号 escape"是不同层面的陷阱（4-6 是字符串字面量解析，4-8 是命令参数解析）
- **规避方案**：
  1. **用 `[regex]::new()` + `.Replace()` 完全绕过命令解析器**：
     ```powershell
     $re = [regex]::new('\!\[\]\(' + [regex]::Escape($folderName) + '/([^)]+)\)')
     $newContent = $re.Replace($content, '![[' + $newFolderName + '/$1]]')
     ```
     - `[regex]::new()` 显式构造正则对象，模式字符串作为 **.NET 构造参数**传递，不受 PowerShell 命令解析影响
     - `.Replace()` 是 .NET 实例方法，两个参数都是方法调用实参，同样不受 PowerShell 命令解析影响
  2. **字面量替换优先用 `.Replace()` 而非 `-replace`**：当替换源是"确定的字面路径/字面字段名"而非通配模式时，`$content.Replace('oldLiteral', 'newLiteral')` 最安全（无需考虑正则元字符，也绕过 `-replace` 参数解析）
  3. **替换字符串中的 `$1` 变量陷阱**：在 `.Replace()` 方法中 `$1` 是正则替换占位符；但在 PowerShell 双引号 `"$1"` 中会被解析为空变量（未定义）。确保替换字符串用单引号包裹，或写成 `'$1'` 避免 PowerShell 变量展开
- **判据**：`.Replace()` vs `-replace` 选用规则（与 4-6 末尾规则互补）：
  - 替换源是「字面路径 / 字面 wikilink / 字面字段名」→ `.Replace()`（无需 escape，无参数解析风险）
  - 替换源是「带通配的正则模式」→ `[regex]::new($pattern).Replace()`（绕过命令解析器）
  - **避免**：动态构造的模式/替换串直接传给 `-replace`（参数解析风险高）
- **案例**：2026-06-05 批量修复 14,779 处图片链接时，`-replace` 在 82 个文件上因替换字符串含空格/括号而连续失败；改用 `[regex]::new() + .Replace()` 后一次通过。同一次修复中，13 个文件因"从文件系统读取当前文件夹名"替换 md 内容中的旧名，导致替换率为 0；修正为"从 md 内容 regex 提取旧名后再替换"后成功（见 [[经验沉淀]] 2-7）

### 4-9 PowerShell `-match` 在 YAML frontmatter 中的 `^` 锚点陷阱 🔥
- **现象**：用 `$content -match '^topic_type:'` 检查 YAML frontmatter 中是否存在某字段时，始终返回 false，即使该字段确实存在。导致批量脚本误判"缺失"，重复添加 `topic_type` 行 → 12 个文件全部出现重复字段。
- **原因**：PowerShell 的 `-match` 操作符在**单行模式**（默认）下，`^` 锚点匹配的是**整个字符串的开头**，而非每一行的开头。YAML frontmatter 中的 `topic_type` 在第 4-5 行，`^topic_type:` 只匹配第 1 行，永远撞不到。
- **规避方案**：
  1. **检测 YAML 字段存在性**时，必须加 `(?m)` 修饰符：`$content -match '(?m)^topic_type:'`
  2. **批量修复前先做小样本测试**：取 1 个文件验证 match/replace 逻辑正确，再扩到全部
  3. **写入后做字节级验证**：不依赖同一正则再次 match，而是用独立方法（如 `[regex]::Matches($content, '(?m)^topic_type:').Count`）确认条数
- **角色界定**：
  - `-match` 做**检测** → 必须带 `(?m)`
  - `-replace` 做**替换** → 也必须带 `(?m)`（若替换目标在 non-first-line）
  - `[regex]::Matches().Count` 做**计数验证** → 同样带 `(?m)`
- **案例**：2026-06-06 Round 3 质量审核阶段，用 `$content -match '^topic_type:'` 检查 12 个专题页 → 全部误判为缺失 → 批量追加 `topic_type` → 全部出现重复行。后续用 `-match '(?m)^topic_type:'` 发现真相，逐文件删除重复行修复。

### 5-3 wikilink 单向（A → B 但 B 不指 A）
- **现象**：A 笔记引用了 B，但 B 笔记没反向引用 A，导致 B 视图下找不回 A
- **原因**：Obsidian 反向链接面板只显示已有的 wikilink；A→B 不蕴含 B→A，B 视图下看不到 A 等于知识网络的一条边变成单向，导航与图谱算法都受损
- **规避方案**：
  1. 利用 Obsidian 的「反向链接」面板核查
  2. 交叉链接 Agent（Agent 5）的核心职责
  3. 关键 KP 在「## 关联」段显式列出双向链接
- **案例**：APiIC Phase 3 收尾时大量补双向链接

### 5-4 同一概念在多个 KP 中重复定义
- **现象**：「氧化加成」在 [[决赛11-金属有机化学]]、氧化加成、18电子规则 中各写一遍，三处定义略有差异
- **原因**：知识库的'权威性'靠单点真相维持；多处定义会随时间漂移，更新一处忘了同步其他处则知识库整体一致性退化
- **规避方案**：
  1. 一个概念一个权威定义页（canonical KP），其他位置用 wikilink 引用
  2. KP 合并 Agent（Agent 8）定期审计

### 5-5 历史快照里的「Top 5」过期
- **现象**：[[状态摘要]] 中的「高优 Follow-up Top 5」一周不动，最近完成的项目还在列表里
- **原因**：手维护的'快照表'没有自动同步机制，时间一长就成历史化石；Phase 4 是唯一硬触发刷新的钩子（v1.2 已用 Dataview 自动化）
- **规避方案**：Phase 4 归档时强制更新这一段，已完成的删除/划掉，新出现的补上
### 5-6 KP 层级判断三问（候选 KP 三选一框架）🔥
- **现象**：人工分诊候选 KP 时，「删除 / 挂载到母 KP / 升格为独立 KP」三选一容易出现「过激删除」——仅看「字面是否被既有 KP 提及」就判为删除，丢失独立的知识价值。
- **触发**：2026-05-17 第五轮 D4 分类时把「羰基α位烷基化 / 有机反应的电子观点 / 芳香体系中吸电子基作用 / 氨基酸的酸碱性」都判为重复删除；用户挑战后用三问框架重审，4/5 应升格而非删除。
- **三问框架**：
  - **Q1 总论价值**：是否覆盖一类反应/思维框架？（如「羰基α位烷基化」是 Aldol/Claisen/Knoevenagel/Mannich 的母概念）
  - **Q2 视角差异**：是否提供与既有 KP 不同的切入点？（如「有机反应的电子观点」是哲学层视角，与「亲核体亲电体」分类层不同）
  - **Q3 同义换字**：是否仅是同义概念的不同表达？（如「取代消除竞争」=「SN1/SN2 vs E1/E2 竞争」字面同义）
- **决策规则**：
  - **任意 Q1 或 Q2 阳性 → 升格独立 KP**（D3，标记为「待创建」）
  - **只有 Q3 阳性 → 删除重复**（D4，从待办池移除）
  - **Q1/Q2/Q3 全阴性，但内容属母 KP 子段落 → 挂载**（D1，wikilink 指向母 KP）
- **规避方案**：
  1. 候选 KP 三选一时**先填三问表**再下结论
  2. 与既有 KP 字面相似时**不直接判重复**，先问「能否在 5 个反应中作总论？」「能否提供新视角？」
  3. 删除前再问一次：「未来备课时这个独立的标题是否还会冒出来？」如果会，就升格而不是删除
- **底层逻辑**：分诊框架（D1/D2/D3/D4）是表面规则，真正的判据是「独立知识价值」——三问就是这个独立价值的具象化测试
- **条件变体合并 vs 机理异质拆分（补充判据）**：通过了三问框架后,若候选与既有 KP 还有部分重叠,再用此判据做最后切分:
  - **条件变体合并**：同一机理的不同条件变体（温度/催化剂/溶剂的细调）→ **合并到一个 KP**,作为子段落或对比表
    - 示例：[[Prevost反应]] vs [[Woodward反应]]（同为银盐双羟化,反式 vs 顺式变体）→ 合并为 [[双羟基化反应]] 总论 + 表格对比
  - **机理异质拆分**：机理本质不同（碳正/碳负/自由基/协同 路径不同）→ **拆为独立 KP**
    - 示例：[[Lindlar催化]]（H₂ + Pd 中毒催化剂,顺式半还原,金属表面催化）vs [[Na-NH3液氨还原]]（单电子转移,反式半还原,自由基阴离子机理）→ 两个独立 KP,因为机理本质不同
  - **判据**：试着写一段"机理"——如果用同一段能说清就合并,如果必须分成两段才说清就拆分
  - **下游 Agent 价值**：Agent 8（合并/弃用）做去重决策时直接用此判据,不再依赖人工模糊判断
- **案例**：[[工作日志/2026-05-17]] 第五/六轮（D4 原 5 处中 4 处回流到 D3 升格清单）

### 5-7 分诊前未搜索既有 KP 导致重复创建风险
- **现象**：D3 升格清单中的「氨基酸的酸碱性」准备创建时，发现已有 [[氨基酸与等电点]]（v1.1，已填充），内容完全覆盖酸碱性讨论
- **原因**：桶 C 分诊时只看 redlink 名称，没搜现有 KP 的 aliases/title 是否已覆盖同一概念；不同 wording（「酸碱性」vs「等电点」） masking 了实质重复
- **规避方案**：
  1. **升格/创建前先 Glob 搜索**：`知识点/**/*酸*` 或 `Grep "酸碱性" 03-知识点/`
  2. **检查现有 KP 的 aliases**：很多 KP 用 aliases 覆盖同义名称
  3. **三秒规则**：任何 D3 候选在创建前，花 3 秒搜一下现有库是否已有覆盖
- **案例**：2026-05-17 第八轮，「氨基酸的酸碱性」改为挂载到 [[氨基酸与等电点]]，避免重复

### 5-8 批量 KP 创建的资料匹配判据（三档评估法）
- **现象**：用户要求"把D3的KP填充一下，没有资料的说一声"，需要快速对 11 个候选做资料-可行性评估
- **原因**：若没标准化判据，每个候选的评估时间不可控，容易过深（读完整章）或过浅（漏掉关键支撑）
- **三档评估法**（2026-05-17 第八轮沉淀）：
  - **A 档「集中论述」**：资料文件中有 ≥1 个独立段落/小节专门讨论该概念 → **直接填充**
  - **B 档「分散提及」**：资料中多处顺带提及，无集中论述 → **视情况**：若概念是总论型（如「反应选择性的来源」），分散提及不足以支撑独立 KP；若概念是具体反应（如「鏻盐」），分散提及 + 通用知识可勉强填充
  - **C 档「无资料」**：资料文件 grep 0 命中，或只有标题级提及 → **阻塞，等资料导入**
- **补充**：B 档的边界最模糊，建议用「KP 层级判断三问」（沉淀 5-6）辅助：若候选是 Q1/Q2 阳性（总论/视角），则 B 档不足；若 Q3 阳性（具体概念），则 B 档 + 通用知识可填充
- **案例**：2026-05-17 第八轮
  - A 档（7 个）：σ络合物 / 协同反应 / 羰基α位烷基化 / 芳香体系中吸电子基作用 / 有机分子的几何构型 / 轨道观点下的稳定性 / 钛铬锰铁钴镍铜锌基础元素化学
  - B 档（无独立成 KP）：鏻盐（Wittig 中顺带提及）
  - C 档（3 个阻塞）：鏻盐 / 反应选择性的来源 / 有机反应的电子观点

### 5-9 DataviewJS 缺口扫描的 aliases 盲区 🔥
- **现象**：知识库待补充清单的 DataviewJS 自动化缺口扫描显示大量"缺失KP"，但人工核实后发现很多概念已被现有KP的aliases或内容覆盖（如`三苯基膦`→`膦化合物.md` aliases含"三苯基膦"；`活化基与钝化基`→`SEAr定位规则.md` aliases含"活化基""钝化基"）
- **根因**：当前 DataviewJS 脚本只检查文件名是否匹配（`app.vault.getAbstractFileByPath` 和 basename 子串匹配），完全不读取 frontmatter 的 `aliases` 字段。因此只要考纲条目中的wikilink名称≠KP文件名，即使aliases完美覆盖，也会被计为缺失。
- **触发**：2026-05-17 第九轮假阳性清洗，18个"真正缺失"中14个是aliases盲区导致的假阳性。
- **规避方案**：
  1. **短期（手动挂载）**：发现假阳性后，将考纲条目中的 `[[概念名]]` 改为 `[[实际KP|概念名]]`（D1挂载操作），这是当前最有效的方法。
  2. **中期（脚本增强）**：在缺口扫描脚本中加入 aliases 读取逻辑——先遍历所有KP文件读取 frontmatter aliases，构建"别名→文件名"映射表，再进行存在性检查。
  3. **长期（命名规范）**：新建KP时尽量让文件名覆盖考纲条目中可能引用的常见名称，或在aliases中写入考纲常用名。
- **案例**：`膦化合物.md` 的 aliases 含 "鏻盐" 和 "三苯基膦"，但考纲条目 `48-膦化合物.md` 中写的是 `[[鏻盐]]` 和 `[[三苯基膦]]`，DataviewJS无法识别，导致假阳性。修复：将考纲条目中的wikilink改为 `[[膦化合物|鏻盐]]` 和 `[[膦化合物|三苯基膦]]`。
- **批量审计脚本**：已建立 PowerShell 脚本（见 [[工作日志/2026-05-17]] 第九轮），可离线模拟DataviewJS扫描并输出"真正缺失"vs"假性缺失"清单,用于快速审计。

### 5-10 光谱型 KP 的构建模式（总论-分论双层架构）🔥
- **现象**：批量填充某类反应（如各类氧化剂 / 还原剂 / 加成试剂）时,每个具体试剂 KP 都把"选择性来源 / 适用范围 / 与同类对比"再写一遍。后期想看「这一类反应整体怎么选」却找不到一个总论入口,被迫横向对照 10+ 个分论 KP。
- **触发**：2026-05-17 桶 C D3 升格阶段,「σ络合物 / 协同反应 / 羰基α位烷基化 / 芳香体系中吸电子基作用」等概念升格时反复纠结「这是不是已经在某个具体反应 KP 里写过了」——根因是缺少"光谱型总论 KP"作为信息汇聚节点
- **原因**：知识库默认按「具体反应名」组织（[[硼氢化反应]] / [[臭氧化反应]] / ...）,缺少「按反应族 / 思维框架」组织的总论层;分论 KP 各自完整但合不成全景图
- **规避方案**（总论-分论双层架构）：
  - **总论型 KP**：覆盖一类反应/思维框架,写「选择性光谱」「电子/立体效应坐标系」「与同类的边界」,**不重复试剂细节**
    - 示例：[[氧化还原反应]]（总论）只立框架:氧化态变化的判定 / 半反应法 / 标准电极电势序 / 常见氧化剂还原剂分类树
    - 具体试剂如 [[KMnO4氧化]] / [[OsO4双羟化]] / [[Cr(VI)氧化]] 在分论 KP 中详述
  - **分论型 KP**：覆盖具体反应/试剂,写「机理 / 条件 / 立体化学 / 区域选择性 / 典型案例」,**显式 wikilink 到总论**
  - **何时立总论**：当 ≥3 个分论 KP 共享同一框架视角（如「亲核加成的电子学解释」「过渡金属催化的氧化加成 - 迁移插入 - 还原消除循环」）时
- **下游 Agent 价值**：
  - Agent 1（提炼）：遇到总论型概念（如"芳香体系中吸电子基作用"）不再纠结要不要拆,直接升格为光谱型 KP
  - Agent 5（链接）：用「分论 → 总论」反向链接确保任何具体反应都能溯源到光谱
  - Agent 10（备课）：先讲总论建框架,再讲分论填细节;备课大纲天然顺着总论树展开
- **判据**：用沉淀 5-6 的三问框架 Q1（总论价值）阳性 → 升格为光谱型总论;纯具体反应 / Q3 阳性 → 留在分论
- **底层逻辑**：知识网络需要"星型枢纽"作为信息聚合点,否则只能在叶子节点间横向跳;总论 KP 就是这些枢纽,让用户/AI 都能"看一层就理解一片"
- **案例**：[[氧化还原反应]]（光谱型）vs [[硼氢化反应]]/[[臭氧化反应]]（分论）;[[加成反应]] 总论 vs 各具体加成 KP;2026-05-17 第八轮 [[有机分子的几何构型]]/[[轨道观点下的稳定性]] 升格为视角型总论 KP

### 5-11 专题页质量审核应按轮次容忍结构差异 🔥
- **现象**：审核 Round 3 有机专题页时，发现它们比 Round 1/2 多了「专题定位」「第三轮总流程」「主干内容分支」「常见误区与纠偏」「与备课框架对应」等额外章节。若按 Round 1 的"标准 8 章"一刀切，会误判 12 个专题全部"结构异常"。
- **原因**：Round 3（提高班/竞赛冲刺）的专题天然需要比 Round 1（基础班）更多的"元认知层"——学生需要知道这个专题在整个体系中的位置、与前后专题的衔接、常见的命题陷阱。这是教学深度的体现，不是结构违规。
- **规避方案**：
  1. **审核基线**：8 个必需章节（定位/核心结论/对比表格/解题套路/机理拆解/典型例题/关联知识点/关联题型/相关真题）是底线，额外章节不扣分
  2. **逐轮次差异化审核表**：
     - Round 1（基础班）：8 章底线，容忍 ±1 章
     - Round 2（提高班）：8 章底线 + 「常见误区」推荐，容忍 ±2 章
     - Round 3（冲刺班）：8 章底线 + 「专题定位」「总流程」「主干分支」「误区」「框架链接」全部推荐，容忍 ±5 章
  3. **审核结论用词**：「结构完整」= 底线章节全有；「结构精品」= 底线 + 推荐章节全有
- **案例**：2026-06-06 Round 3 质量审核时，周环反应专题有 13 个章节（标准 8 + 5 个 Round 3 特有章节），审核结论为"结构完整，含 Round 3 增强章节"，而非"结构异常"。

### 5-12 README 配图表严重失真（手维护表的典型风险）
- **现象**：[[04-课件/学生讲义/README.md]] 的配图统计表与实际情况严重不符：7 份已有配图的讲义被错误列入"无配图"清单，1 份讲义表记 2 张但实际 0 张。"无配图清单"从真实的 5 份膨胀到 13 份（含 8 份误差）。
- **原因**：README 的配图表是手工维护的计数表，每次配图战役后需要手动更新行数和数字，存在以下脆弱点：
  1. 新增配图时只加新行，忘了检查已有行是否准确
  2. "无配图清单"是减法逻辑，但只有在确认"有配图"时才移除，容易积累过期条目
  3. 数字驱动（写"2 张"）比文件驱动（实际读文件确认）更容易漂移
- **规避方案**：
  1. 每次配图后强制做全量 Grep 校验：`Grep "!\[\[media/" 04-课件/学生讲义/` 获取实际配图数，与 README 表逐行对比
  2. "无配图清单"改为反向推导：从讲义总数中减去实际有图的，而非手工累加"无图"的
  3. 未来可在讲义模板加 frontmatter 字段后用 Dataview 自动生成配图表（见 [[经验沉淀#1-8 学生讲义缺乏图片追踪 frontmatter 字段 🔥]]）
- **案例**：2026-06-24 全面复查时发现 7 份 Batch 1 讲义（原子结构/元素周期表/分子结构/气体基础/溶液和胶体/化学平衡/水中的几种平衡）已有配图但被错误计入"无配图"清单；元素化学深度表记 2 张/实际 0 张。修正后已配图 49→54 份，无配图 13→5 份。

## 六、并行 Agent 协作类

### 6-1 同一文件被两个 Agent 写 → 脏数据 🔥
- **现象**：并行启动两个 Agent，都修改了 元素周期表，第二个完成后覆盖第一个的改动
- **原因**：文件系统不提供并发写入冲突检测，第二个 Agent 完成后的覆盖会丢失第一个 Agent 的所有改动（last-write-wins 语义）
- **规避方案**：
  1. **强约束**：Plan Agent（[[Agent标准作业流程]] Phase 1）规划任务时必须保证不同 Agent 的文件范围不交叠
  2. 文件级锁：单个 KP 由单个 Agent 全权负责
  3. 共享文件（[[状态摘要]]、[[工作日志]]）只能由 Phase 4 归档 Agent 修改
- **案例**：2026-04 早期并行批次踩过，必须避免

### 6-2 子 Agent 报告不一致
- **现象**：Plan Agent 拆出 3 个子任务，3 个子 Agent 报告的「完成」标准不统一（有的算半成品，有的算完美）
- **原因**：每个 Agent 对'完成'的理解会分化（半成品/最低标准/完美），如果 Plan 端没有统一 DoD（完成定义），子 Agent 各自决定自己满意了
- **规避方案**：
  1. Plan 时给每个子 Agent 写「DoD」（Definition of Done，完成定义）
  2. Phase 3 验证时统一过一遍 DoD 清单
- **案例**：[[Agent提示词-知识库构建]] 各 Agent 的「后置检查清单」就是 DoD

### 6-3 子 Agent 上下文不够丰富
- **现象**：Plan Agent 给子 Agent 的 prompt 太简略（'填充这 5 个 KP'），子 Agent 不知道用什么模板/什么标准
- **原因**：子 Agent 是'无记忆冷启动'——它没看过本会话的讨论上下文，prompt 是它的全部输入；prompt 简陋等于让子 Agent 在信息真空中执行
- **规避方案**：每个子 Agent prompt 必须含：
  1. 任务范围（哪些文件）
  2. 引用的模板（[[模板-知识点]] vs [[模板-课件大纲]]）
  3. DoD（完成定义）
  4. 不做什么（边界条件）
- **案例**：本会话开局重新设计 prompt 框架

### 6-4 串行依赖被错当成并行
- **现象**：「填充 KP A → 在 KP B 中加引用 A」被启了两个并行 Agent，B 启动时 A 还没写完
- **原因**：依赖关系（B 需要 A 完成）是逻辑约束，并行启动等于无视约束；启动时 A 未完成则 B 拿不到所需数据/前置 KP，要么报错要么基于占位状态执行
- **规避方案**：Phase 1 计划时显式画依赖图（DAG），有依赖的串行启动

### 6-5 先 MOC 同步、再 KP 落地（Tier 1+2 工作流）
- **现象**：分批补 KP 时按"挑一个 → 填一个 → 挂载一次"的顺序推进,结果:
  - 母 KP / 总览 / 考纲条目 等"账本类"文件被改 5+ 次（每次新建一个 KP 都要补反向引用）
  - 工作日志 H2 标题反复刷新（每填一个 KP 写一次"路线 X 第 Y 个"）
  - Agent 上下文反复加载同一个母 KP（每次都从开头读到末尾）
- **原因**：把"账本"（MOC / 总览 / 考纲条目 / 母 KP）和"账目"（具体 KP）混在一个循环里写。账本是低频高扇出（少改但波及大）,账目是高频低扇出（多改但局部影响）;混在一起改让账本被反复 dirty
- **规避方案**（两阶段工作流）：
  - **Tier 1 先做账本同步**（一次性批改）：
    1. 列出本批次所有要补的 KP 清单
    2. 一次性更新母 KP / 总览 / 考纲条目,把所有目标 KP 的 wikilink 占位先写好（指向尚未创建的红链是 OK 的,Obsidian 会以红链显示但不影响占位功能）
    3. 一次性更新 [[状态摘要]] 数据锚点段（如「桶 X 进度 N/M」）
  - **Tier 2 再做 KP 落地**（批量并行 / 串行）：
    4. 按计划填充 KP 内容,每个 KP 完成时只动它本身（账本已不需要再改）
    5. 收尾时 Phase 4 一次性归档
- **下游 Agent 价值**：
  - Plan Agent（Phase 1）：天然分两段计划,Tier 1 / Tier 2 串行
  - 并行 Agent（6-1 文件级锁）：Tier 2 阶段每个 KP 文件由单 Agent 全权负责,账本已稳态不再争抢
  - Phase 4 归档：账本改动一次即可写一次日志,避免分散在每个 KP 完成时重复刷新
- **判据**：本批次涉及 ≥3 个新 KP 时强制走 Tier 1+2;单个 KP 改动可直接做
- **底层逻辑**：账本同步是"读多写少"操作,KP 落地是"读少写多"操作;用 Tier 分阶段把两类操作分离,让每类工具调用都达到最优批量
- **案例**：2026-05-17 桶 A/B/C 收尾时,先一次性更新 [[syllabus_gap_v3_categorized]] 与考纲条目（Tier 1）,再批量填充 D3 升格 KP（Tier 2）;对比早期"挑一个填一个"的方式,重复读取次数下降 ~70%

## 七、批次/规模类

### 7-1 单次会话试图填充 30+ KP
- **现象**：上下文超载、Agent 性能下降、错误率飙升
- **原因**：上下文窗口和注意力都有上限；任务量超过临界点后 Agent 性能曲线断崖下降（漏段、重复、风格漂移），不是线性退化
- **规避方案**：单批次上限 ~10 KP，超过的拆批次
- **案例**：化学原理 73 个分了 5 批做完

### 7-2 批量 Edit 同一字段时格式漂移
- **现象**：批量给 70 个 KP 加 `aliases: []`，写到第 50 个时不小心写成 `alias:`（少了 es）
- **原因**：人手重打很容易引入笔误（`alias` 漏 es / 全角冒号）；70 次重打产生 ≥1 次错的概率近 100%（Poisson 分布）
- **规避方案**：
  1. 写一次 Edit 模板存好，复制粘贴而不重新打字
  2. 批次结束后用 Grep 抽样核对：`Grep "^aliases:" 03-知识点/ -c`
- **案例**：分析化学批次 17 个 KP 加字段时

### 7-3 工作日志一次写太长
- **现象**：归档时写了 5000+ 字，重点淹没
- **原因**：归档目的是'未来快速回看与抽取'，不是'完整事件日记'；过长内容稀释信号密度，关键信息淹没在叙事流里
- **规避方案**：[[工作日志]] 强约束 6 段（任务/范围/计划/执行/验证/Follow-up），每段控制行数

### 7-4 Phase 0 读太多文件浪费 token
- **现象**：每次 Agent 开会话都要读 5 个核心文件（SOP/模板/索引/工作日志/经验沉淀），开销大
- **原因**：5 个核心文件加起来 ~10000+ token，每次新会话都加载等于把启动开销线性放大；状态摘要的核心价值就是'压缩 5 → 1'
- **规避方案**：v1.2 SOP 引入 [[状态摘要]] 单文件入口，仅在跨模块大批量任务时才读全部 5 个
- **案例**：本次重构的核心动机

## 八、多源资料整合与备课类

### 8-1 网课资料文件名与内容严重错位 🔥
- **现象**：按文件名直接归类备课，结果 `10.化学平衡（一）_笔记` 实际讲热力学，`12.热力学_笔记` 实际讲题解+电化学，`02.分子结构（一）_笔记` 实际为原子结构习题课
- **原因**：网课OCR笔记由学生上传，文件名常按课程表填写，与实际录制内容脱节；若不核实正文，备课大纲会链接到错误内容
- **规避方案**：
  1. **文件标题错位检查为强制步骤**：Agent 1A 处理网课资料时，必须扫描正文前 20% 确认实际内容，与文件名比对
  2. 在提炼文件顶部用 ⚠️ 醒目标注所有标题错位/内容混杂的文件
  3. 建立"文件错位索引"（见 `[[周坤无机新课-知识库自用授课顺序总表]]` §三）
- **案例**：周坤无机化学 Batch 1~4 共发现 7 处严重错位，若按文件名映射会导致授课顺序完全错乱

### 8-2 同一概念因名字不同重复建 KP
- **现象**：周坤回流目标列出「分裂能」「Pauling规则」「晶体缺陷」「配合物命名」，Agent准备新建4个KP框架
- **原因**：知识库中这些概念已作为子内容存在于其他KP（分裂能→[[晶体场理论]] aliases；Pauling规则→[[离子晶体]] §二；晶体缺陷→[[非化学计量化合物]] tags；配合物命名→[[配合物]] problem_types），但未以独立文件名出现
- **规避方案**：
  1. **回流前强制搜索**：`Grep` 目标关键词 + `Read` 确认内容覆盖度
  2. 检查现有KP的 `aliases`、`tags`、`related`、`problem_types` 是否已含目标概念
  3. **只有确实不存在任何覆盖时才新建KP**；否则将资产回流到已有KP的对应段落
- **案例**：2026-06-01 周坤17个回流目标核查，13个为已有独立KP，4个为已有KP子内容，全部无需新建

### 8-3 多套网课资料导致备课顺序混乱
- **现象**：同时参考周坤和质心两套网课，周坤先讲原子结构再讲分子结构，质心先讲热力学再讲平衡；Agent备课时把两套顺序混在一起，产出一份前后矛盾的备课大纲
- **原因**：不同老师的讲授顺序是为各自课程设计的，不一定适合直接拼接；没有统一的主干顺序，备课大纲会变成"周坤第一节+质心第二节"的缝合怪
- **规避方案**：
  1. **顺序层自己定**：建立统一授课阶段表（如10阶段：原子→分子→晶体→配合物→气体→热力学→平衡→酸碱→电化学→定量），所有资料往这张表映射
  2. **传授层借鉴人**：把不同老师的切入方式、反例、口诀记录到同一KP的「多源教学策略」区，备课时按学情选择
  3. 备课Agent 10 在阶段1.5必须读取 `[[00-首页/方法论-多源教学资料整合]]` 和 `[[周坤无机新课-知识库自用授课顺序总表]]`
- **案例**：`[[00-首页/方法论-多源教学资料整合]]` 已建立两层分离法和四步操作流程

### 7-5 拆批次时跨批次的「整体感」丢失
- **现象**：化学原理分 5 批做完，第 5 批做完时回头看第 1 批的 KP 风格已经漂移
- **原因**：每批次结束时记忆会衰减；下一批次开始时 Agent 不会主动回看上一批次的具体写法，只看模板和上下文，于是风格漂移
- **规避方案**：
  1. 第 1 批必须建「样板 KP」，后续批次以样板为基准
  2. 全部完成后做一次「批次间一致性扫描」（Agent 9 健康检查）
- **案例**：[[模板-知识点]] v1.1 的「📚 标杆样板」段就是为此

## 八、外部资料导入类

### 8-1 PDF 复制内容混入软回车 / 不可见字符 🔥
- **现象**：从 PDF/网页拷贝的段落，看似正常，实际每行末尾或词间夹杂不可见字符，Markdown 渲染断行错乱、wikilink 失效、Grep 模式匹配莫名失败
- **常见元凶**（按踩坑频率排序）：
  - `U+000B` 垂直制表符（VT，PDF 软回车主犯）
  - `U+00A0` 不间断空格（NBSP，常见于 LaTeX/页眉复制）
  - `U+0085` 下一行（NEL）
  - `U+2028` 行分隔 / `U+2029` 段分隔
  - `U+200B` 零宽空格（ZWSP，复制网页高发）
- **原因**：PDF 内部用绝对定位排版，复制时把视觉换行翻译为软换行/不间断空格；OCR/网页同理。Obsidian Markdown 渲染按行分段，遇到这些字符就错位
- **规避方案**：
  1. 粘贴后立即扫描：`Grep -P "[\x{000B}\x{00A0}\x{0085}\x{2028}\x{2029}\x{200B}]" path/to/file.md`
  2. PowerShell 批量清洗（保留正常换行）：
     ```powershell
     $t = [IO.File]::ReadAllText($path,[Text.UTF8Encoding]::new($false))
     $t = $t -replace "[  ]", "`n" -replace "[ ​]", " "
     [IO.File]::WriteAllText($path, $t, [Text.UTF8Encoding]::new($false))
     ```
  3. 推荐流程：粘贴前先经 VS Code「另存为 → UTF-8 不带 BOM」洗一遍，再进 Obsidian
- **案例**：APiIC 资料导入早期；本条原文（2026-05-06 版）描述里都嵌过 `U+000B`+`U+00A0`，2026-05-10 修订时清理；ABOC mineru OCR 输出（详见 8-4）

### 8-2 外部资料 KP 重复创建 / subject 错配 🔥
- **现象**：
  - **场景 A**：从两份资料分两次导入同一概念，建出同名 KP（如 [[Lawesson试剂]] 与 Lawesson 试剂）
  - **场景 B**：早期归为某 subject 的 KP（如「立体电子效应」），后续发现应独立成专题，需要迁移而非新建（β-硅基效应曾踩过此坑）
- **原因**：A 是导入端没查重；B 是知识体系演化（细化/分形）后老归类不再适用，但移动文件会断链所以容易拖延
- **规避方案**：
  1. **导入前查重**：`Grep "<概念名>"` + 别名变体（带空格/连字符/英文别称）扫一遍
  2. **场景 A 处理**：建 alias 不是建新 KP — 把所有写法（含变体）写入老 KP 的 `aliases:` 字段
  3. **场景 B 处理（Plan B：重命名 + 迁移）**：
     - 用 Obsidian 内置「重命名」自动更新所有反向引用（不要直接改文件名）
     - 移动文件到正确 subject 目录后，更新 frontmatter `subject` / `parent_overview` / `parent_module`
     - 在母 KP（原归属）中保留指向新 KP 的 wikilink，并在 `## 八、相关知识` 中说明「已迁移为独立 KP」
     - 同步更新 [[状态摘要]] 库存表（自动 Dataview 重算无需手工）
- **案例**：[[Lawesson试剂]] aliases（场景 A）；Phase C 第二批 [[β-硅基效应]] 从「立体电子效应」子项迁出（场景 B）

### 8-3 外部资料引用无章节 / 行号 / 页码定位 🔥
- **现象**：KP 里只写「来自 APiIC」「见 ABOC」，回溯具体段落需要重读全章
- **原因**：导入时图省事；下游用户（备课/答疑 Agent）拿到 KP 想验证一句话出处时无从下手
- **规避方案**：引用必须三层级精度，按资料类型选用：
  | 资料类型 | 必需精度 | 示例 |
  |---|---|---|
  | 章节明确的教材 / 论文 | §x.y / §x.y.z | `§6.14`、`§2.10.2` |
  | OCR 转录（如 mineru 输出） | §x.y + 行号 | `§2.7（行 1688）` |
  | 纸质书 / 影印 PDF | §x.y + 页码 | `§4.3（p. 127）` |
  | 论文 | DOI + 图/表号 | `Levin 2021, Fig. 3` |
- **写在哪**：KP 的 `## 十五、资料出处` 或 `## 八、相关知识 / 已关联外部资料` 段
- **案例**：[[决赛11-金属有机化学]] 「八、已关联外部资料」（§x.y 范例）；[[端基效应]] §十五（多源混合范例）；[[提炼-ABOC-第2章-基本反应]] §十踩坑回流（§+行号 范例）

### 8-4 mineru OCR 对 LaTeX / 化学式 / SN1-SN2 表达式的破坏 🔥
- **现象**：
  - **LaTeX 上下标错乱**：`Na + H _ {2} \mathrm{O}`（下标位置混乱、空格混入）
  - **化学式下标被吞**：`Br _ {2}` → 部分场景脱化为 `Br2`
  - **SN 表达式碎片化**（↵ 代表原文在此换行）：`$\pmb{\mathsf{S}}_{\mathsf{N}}\pmb{1} ↵ $`、`$\$ 12 ↵ $`、`$\boldsymbol{S}_{N}1$`（三种碎片均应为 `$\mathrm{S_N1}$`）
  - **公式跨段**：`$\Delta G^{\ddagger}` 后接换行 + `$` 单独成行（应合并为单行 `$\Delta G^{\ddagger}$`）
- **原因**：mineru OCR 对带空格的下标 + 复合 LaTeX 命令（`\pmb` / `\mathsf` / `\mathrm` / `\boldsymbol`）处理不佳；公式跨页时 PDF→图→文本三道转换易丢括号
- **规避方案**：
  1. **提炼时全文 Grep 关键词**：`H _ {`、`Br _ {`、`SN1`、`SN2`、`\pmb`、`\mathsf`、`\boldsymbol`
  2. **§六关键图表节用规范 LaTeX 重写**：`$\mathrm{H_2O}$`、`$\mathrm{Br_2}$`、`$\mathrm{S_N1}$` / `$\mathrm{S_N2}$`
  3. **机理图与势能图必须回原 PDF 校对**（OCR 对图内文字识别率低）
  4. 提炼笔记的 `## 十、踩坑回流` 段把这些行号留作钩子（让下游 KP 知道哪些点需要回查）
- **案例**：[[提炼-ABOC-第2章-基本反应]] §2.7（行 1688 LaTeX 上下标）/ §2.8（化学式下标）/ §2.9（SN 表达式碎片）多次踩坑

## 九、编码/字符集类

### 9-1 批量脚本未显式指定 UTF-8 导致中文正文全军覆没 🔥
- **现象**：批量生成/导入后，frontmatter 看似正常，但正文大量中文字符变成 `�`（UTF-8 替换字符）或 `?`。Obsidian 中阅读时满屏乱码，且无法通过简单替换恢复——因为原始字节已永久丢失。
- **原因**：
  1. **Node.js `fs.writeFileSync` 默认编码陷阱**：Windows 中文系统 locale 默认是 GBK/GB2312，若脚本不写 `{ encoding: 'utf-8' }`，中文字节被以系统默认编码写入
  2. **PowerShell 管道默认编码陷阱**：`Get-Content` / `Set-Content` 默认跟随 `$OutputEncoding`（常为 OEM CP936），管道传输会转码
  3. **PDF→文本提取工具的输出编码与后续处理脚本不一致**：MinerU/OCR 输出 UTF-8，但下游脚本以系统默认编码重新保存，双重转码后彻底损坏
- **规避方案**：
  1. **所有文件写入必须显式声明 UTF-8**：
     ```javascript
     // Node.js
     fs.writeFileSync('file.md', content, { encoding: 'utf-8' });
     ```
     ```powershell
     # PowerShell
     [IO.File]::WriteAllText($path, $content, [Text.UTF8Encoding]::new($false))
     ```
  2. **批量生成后强制做编码审计**（写入后 5 分钟内执行）：
     ```powershell
     Get-ChildItem -Recurse -Filter *.md | ForEach-Object {
       $c = [IO.File]::ReadAllText($_.FullName, [Text.UTF8Encoding]::new($false))
       $bad = ([regex]::Matches($c, "�")).Count
       if ($bad -gt 0) { Write-Output "$($_.Name): $bad 个乱码" }
     }
     ```
  3. **CI 级硬规则**：任何批量写入脚本必须在 header 注释中写明 `// encoding: utf-8`，Code Review 时检查
- **案例**：2026-05-23 全库审计——702 个 KP 中 **331 个**（47%）正文含 `�`/`?` 乱码，总计 **86,837** 个损坏字符，遍布有机化学/无机化学/化学原理三大模块。根因追溯到 MinerU 提取 + batch-create-kp.js 未显式指定 UTF-8。
- **案例**：2026-05-25 第三批有机化学迁移——5.24 刚清零的编码在 122 个 KP 批量写入后又出现 48 个文件（21,960 `�`），Agent 写入环节再次触发同一陷阱。
- **修复成本**：编码损坏不可逆（原始字节已丢失），只能接受现状或从原始 PDF 重新提取。

### 9-2 BOM 头导致跨平台同步后 frontmatter 解析失败
- **现象**：Windows 上创建的 .md 文件在 macOS/Linux 同步后，Obsidian 读不到 frontmatter，或 `type: 知识点` 被识别为正文
- **原因**：`[Text.UTF8Encoding]::new($true)`（带 BOM）写入的文件，在部分编辑器/Obsidian 版本中被解析为 `---` 前多了三个不可见字节，导致 frontmatter 边界识别失败
- **规避方案**：**永远使用不带 BOM 的 UTF-8**：
  - PowerShell：`[Text.UTF8Encoding]::new($false)`（$false = 不带 BOM）
  - Node.js：`fs.writeFileSync` 默认不带 BOM，不要手动加 `\xEF\xBB\xBF`
- **案例**：经验沉淀 4-7 已规范 `.NET` 写法的 `$false` 参数；若忘记写 `$false`，BOM 头混入后 Dataview 查询随机失效

### 9-3 中文字符在正则/Grep 中的编码陷阱
- **现象**：用 `Grep` 搜索中文关键词时，明明文件中有该词却匹配不到；或 PowerShell 正则 `[一-鿿]` 在某些环境下匹配范围异常
- **原因**：
  1. **文件实际编码非 UTF-8**：Grep 按 UTF-8 解析，但文件是 GBK/GB2312，中文字节序列被拆成乱码，无法匹配
  2. **终端字体/代码页限制**：Windows PowerShell 默认代码页 936（GBK），若文件 UTF-8 无 BOM，终端显示正常但底层字节与 Grep 期望不一致
  3. **正则范围写法差异**：`[一-鿿]` 在 JavaScript 有效，在 ripgrep/PowerShell 中需用 `[一-鿿]` 或 `[\x{4e00}-\x{9fff}]`
- **规避方案**：
  1. **统一文件编码**：入库前用 `file -i` 或 VS Code 确认是 UTF-8
  2. **Grep 优先用 ripgrep**：`Grep` 工具内部用 ripgrep，对 UTF-8 支持最好；避免用 PowerShell `Select-String` 做中文匹配
  3. **JavaScript/Node.js 处理中文正则**：统一用 `/[一-鿿]/u`
- **案例**：2026-05-23 扫描乱码时，PowerShell `[regex]::Matches` 与 Node.js 正则结果不一致，最终统一用 Node.js 脚本做统计

### 8-5 术语译名冲突 / OCR 误字
- **现象**：

  3. **OCR 形近字定向 Grep**：导入后 `Grep -E "鎓|锇|腈|胺|醇|酰"` 抽样检查，结合上下文判断
  4. **AI 提炼时自检**：发现某术语在同段出现 ≥2 种写法就停下回查

### 8-8 外部资料目录整理后未同步反向引用 → 批量断链 🔥
- **现象**：把 mineru/ 子目录（如 `mineru/某书章节/images/` → `mineru/某书章节/figures/`）整理或重命名后，整个仓库内引用这些图/文件的 wikilink/markdown 链接集体断链。表面看是「整理后没动其他文件」，实际是 5 类 901 处批量断链（[[工作日志/2026-05-10]] 案例）
- **原因**：外部资料目录的「文件夹移动」与 Obsidian 内置「重命名」（自动更新引用）不同——它在 Obsidian 视角下等同于「删除旧文件 + 新建无关文件」，反向引用追溯机制完全失灵；这是 2-3「重命名后留下断链」的**批量延伸版**
- **规避方案**（整理前先做这套）：
  1. **先 Grep 反向引用清单**：
     ```powershell
     # 同时扫 wikilink embed (![[...]]) / wikilink ([[...]]) / 裸路径
     Get-ChildItem -Recurse -Filter *.md | Select-String -Pattern 'mineru/旧前缀' -AllMatches
     ```
     用「字符串包含」而非纯 wikilink regex（避免漏掉工作日志中的裸路径）
  2. **建替换字典**：`@{ 'mineru/旧前缀' = 'mineru/新前缀'; ... }`（单引号字符串规避反引号 escape — 见 4-6）
  3. **`.Replace()` 字面量替换**（替换源含 `()`、`（）`、`,` 等元字符时，`.Replace()` 比 `-replace` 安全 — 见 4-6 字符串替换 API 选用）
  4. **批量写入**（用 4-7 模式，UTF8NoBom）
  5. **Phase 3 全验证**：抽 1-2 张被引图实地打开（如题库题目），确认链接生效；Dataview 「不存在的引用」面板应为 0
- **统计差异留意**：估算用「字符串包含」匹配（901 处），实际批量替换报告也用 `Before -ne After`；纯 wikilink regex `!\[\[mineru/...` 会漏掉 ~25% 的引用（普通 wikilink + 工作日志裸路径）
- **案例**：2026-05-10 mineru/ 整理后用户报告图片找不到 → 侦察发现 5 类 901 处批量断链 → 单脚本（单引号字典 + `.Replace()`）一次性修复 67 个文件，Phase 3 全验证通过（[[工作日志/2026-05-10]] L1604-1610）

### 8-9 批量移动文件后 Obsidian 索引未刷新 + PowerShell 编码错误导致文件损坏 🔥
- **现象**：
  1. 用 PowerShell `Move-Item` 将 mineru/ 根目录下 30+ 个教材 md 文件及 images 文件夹移入 `03-教材书籍/子文件夹/` 后，用户在 Obsidian 中点击这些文件全部显示"打开失败"。
  2. 同时，用 `Set-Content -Encoding UTF8` 修复 `33届初赛试题解析.md` 和 `36届初赛试题解析.md` 的图片路径时，两个文件的中文内容全部变成乱码（不可逆损坏）。
- **原因**：
  1. **Obsidian 缓存未刷新**：文件被移动后，Obsidian 的文件浏览器和索引仍显示旧位置的缓存。用户看到的是旧文件列表，点击后 Obsidian 找不到实际文件，显示"打开失败"。
  2. **编码处理错误**：`Get-Content` 默认按 UTF-8 读取了实际上是 GBK 编码的 md 文件，产生乱码字符串；随后 `Set-Content -Encoding UTF8` 把乱码永久写入文件，原始中文信息丢失。
- **规避方案**：
  1. **移动文件前**：
     - 先评估是否真的需要移动（vs. 保持原地 + 更新索引/链接）
     - 如需移动，必须先 Grep 所有反向引用（wiki link、书签、快速切换器历史、工作日志中的裸路径），评估断链影响
     - 移动后必须提示用户按 `Ctrl+R` 刷新 Obsidian，或重启应用
  2. **修改文件内容时（PowerShell）**：
     - **绝不**用 `Get-Content` + `Set-Content` 直接修改已有中文内容的文件，除非**先确认文件编码**
     - 先用 `[IO.File]::ReadAllBytes` 检查 BOM，或用 `file` 命令确认编码
     - 如需修改含中文的文件，优先用 Edit 工具（安全、有校验）；如需 PowerShell 批量处理，必须显式指定正确的编码（如 `[Text.Encoding]::GetEncoding("GBK")` 或 `[Text.Encoding]::UTF8`），且修改前先做备份
     - 修改后立即 Read 抽样验证中文内容是否正常
  3. **回滚策略**：
     - 一旦用户报告"打不开"，第一优先级是**回滚到原位**，而不是尝试修复链接或让用户刷新缓存
     - 回滚后立即验证文件数量和路径是否恢复
- **案例**：2026-05-19 mineru/ 教材整理。将 Clayden/ABOC/超分子化学/无机化学/物理化学 等 30+ 文件移入子文件夹后，Obsidian 显示"打开失败"；同时 33届/36届初赛试题解析.md 中文内容被 `Set-Content -Encoding UTF8` 破坏为乱码。最终全部回滚到原位解决。

### 8-10 Obsidian 运行时安全整理文件夹的完整方案（8-9 的正向案例）🔥
- **现象**：需要在 Obsidian 保持打开的状态下，将大量 md 文件及配套 images 文件夹从根目录移入子文件夹，同时保证：
  - 文件在 Obsidian 中始终能打开（不出现"打开失败"）
  - 图片链接不丢失
  - 仓库内所有反向引用（README、data.json、配置指南）同步更新
- **原因**：Obsidian 的文件索引在运行时不会自动感知外部文件系统变更；直接移动会导致缓存中的文件路径失效。同时，MinerU 提取的 md 文件使用相对路径 `filename_images/xxx.jpg` 引用图片，只要保持 md 与 images 文件夹的相对位置不变，链接就不会断。
- **完整方案（七步安全流程）**：
  1. **建子文件夹**：在目标位置预先创建好分类子文件夹
  2. **物理移动**：将 md 文件及其同名的 `_images` 文件夹**成对**移入对应子文件夹（保持相对路径 `md文件名_images/xxx.jpg` 不变）
  3. **原地留桩**：在原文件位置创建同名 stub 文件（YAML frontmatter + 跳转链接），防止 Obsidian 缓存中的文件条目指向空路径。写入时必须使用 `[IO.File]::WriteAllText(path, content, [Text.Encoding]::UTF8)`，**严禁使用 `Set-Content`**
  4. **建索引入口**：在每个子文件夹内创建 `index.md`，用 `[[分段文件名]]` 汇总该书的所有分段，方便快速浏览
  5. **同步反向引用**：批量更新 README.md、配置指南、data.json 等所有指向旧路径的引用。data.json 中 `outputPath` 与实际文件名可能不完全一致，只更新文件名匹配的记录
  6. **验证 + 刷新**：抽检文件内容无乱码、图片为相对路径、md 与 images 一一对应；确认无误后让用户在 Obsidian 中按 `Ctrl+R` 刷新索引
  7. **清理桩文件**：确认 Obsidian 已刷新（侧边栏能看到新文件夹结构）后，删除所有 stub 文件
- **关键判据**：
  - 图片路径是否为相对路径（`![]()` 中不含 `http` 或盘符）
  - stub 文件第一行必须是 `---`（YAML frontmatter），且包含"已迁移"字样
  - 删除 stub 前必须确认 Obsidian 已刷新，否则会出现"打开失败"
- **案例**：2026-05-19 mineru/ 教材整理（8-9 失败后的重试）。将 21 个教材 md + 17 个 images 文件夹安全移入 8 个子文件夹，创建 8 个 index.md，更新 36 条 data.json 记录，全程未出现"打开失败"或乱码。

### 8-11 多源备课不能停在"提炼完成"，必须继续闭环到讲义层与工具层 🔥
- **现象**：网课资料提炼和教学逻辑提炼都做完后，表面上已经"可参考"，但真正备课时仍要在多份提炼稿之间来回翻找，课堂入口不稳定，学生复习材料也偏散。
- **原因**：提炼层解决的是"把资料读懂并结构化"，并不自动等于"可上课"。真正可复用的备课体系至少还差三层：学生可直接使用的讲义层、跨专题复用的工具层、把来源和成品串起来的治理层。
- **规避方案**：
  1. 先做 **来源层**：资料提炼 + 教学逻辑提炼
  2. 再做 **成品层**：专题页 → 备课大纲 → 新授课讲义
  3. 再做 **工具层**：总卡 + 单页工具卡
  4. 最后做 **治理层**：转化跟踪总表 + 讲义开头来源/工具入口

### 8-12 资料提炼回流专题页前必须做「已有内容判重」🔥
- **现象**：把资料提炼中的内容回流到专题页时，若不做差异化对比直接追加，会把专题页中已有的内容重复写入（如双指示剂法、返滴定原理、金属指示剂选择等）。这不仅浪费 token，还造成专题页同一知识点出现两套表述不一致的版本。
- **原因**：专题页在更早的轮次中可能已从其他来源（原始笔记、教材提炼）填充了部分内容。资料提炼回流不是”从零填充”而是”增量补强”，必须先判重再决定追加什么。
- **规避方案**：
  1. **回流前做 Grep 判重**：将要追加的每条内容（例题标题、表格名称、关键概念）在目标专题页中搜索，已存在则跳过或仅补充例题
  2. **三档判重决策**：
     - **已存在且内容等价** → 跳过（只更新 `updated` 字段）
     - **已存在但角度/深度不同** → 追加为”补充视角”段落
     - **不存在** → 正常追加
  3. **回流日志记录**：每份资料提炼回流后，在资料提炼文件的 `## 十、踩坑回流` 追加一条”已回流到 [[目标专题页]]，实际追加 N 条，跳过 M 条”
- **案例**：2026-06-06 B-3 阶段回流分析化学资料到专题页时，Grep 发现容量分析专题已覆盖双指示剂法、氧化还原专题已覆盖 KMnO₄ 法原理 → 只追加了 5 道”尚未被覆盖”的高价值操作细节题（CO₂影响、极弱酸强化、碘量法测铜 KSCN、K₂Cr₂O₇ 铁矿石 H₃PO₄、四舍六入五成双），跳过了已覆盖的 8 道通用题。

### 8-13 专题页「反应机理拆解」章节应选该专题最有代表性的单一机理深拆 🔥
- **现象**：为多个专题页补写「反应机理拆解」章时，容易想把该专题涉及的所有反应都拆一遍。结果要么太长（偏离”一页一机理”的设计初衷），要么太浅（每个都写不深）。
- **原因**：此章节的设计目标是”通过一个完整案例让学生掌握机理拆解的方法论”，不是”穷举该专题所有反应”。拆一个反应拆透了，学生自然会拆同类的；拆五个反应都只写两步，等于一个也没教会。
- **规避方案**：
  1. **选题标准**：选该专题”考查频率最高 + 最容易在考场上拆错”的那一个反应
  2. **三步结构**：机理推导（箭头推动 + 中间体标注）→ 检查表（关键判据清单）→ 一句话总结（方法论抽象）
  3. **每章只拆一个**：其余反应在「典型例题」节以完整题目形式处理

### 8-14 竞赛命题的”给材料学习”题型需要专门的解题方法论训练 🔥
