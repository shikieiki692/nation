# Frontmatter 角色词表

> 本页用于冻结 `00-首页` 系统页常用 `role` 取值，避免后续继续出现“同类页面多个叫法”。
> 若只是判断当前该读什么，仍先走 `[[状态摘要]] → [[活跃任务]] → [[Agent战略方向]]`；本页只解决“页面类型怎么标”的问题。
> 实际挂载效果与漏标情况，见 [[系统页角色索引]]。

---

## 一、使用规则

1. `role` 优先表达“这页在系统中的职责”，不要表达临时阶段或具体内容主题。
2. 同一类页面尽量只保留一个固定叫法，不混用近义词。
3. 若新页面能归入现有词表，就不要新增 `role` 变体。
4. 只有当现有词表无法覆盖新页面职责时，才新增角色词，并顺手回写本页。

---

## 二、`type` 枚举规范

> 从 2026-06-15 起，前台系统层统一按“系统 / 活跃任务卡 / 工作日志”三层枚举收口。
> 旧的 `说明`、`评估`、`方法论`、`课件索引`、`备课框架`、`命题分析`、`日志` 等 `type` 不再作为新口径继续扩散。

| `type` | 用途 | 典型页面 |
|---|---|---|
| `系统` | 入口、规则、索引、框架、说明、分析、方法页等所有系统/半系统页面 | `00-首页/` 主入口页、`备课思路/` 框架页、`02-数据库/` 入口页、`06-学生侧材料/` README |
| `活跃任务卡` | 短期执行层任务卡 | `00-首页/活跃任务/任务卡-*` |
| `工作日志` | 按日期记录过程、证据、结论的日志页 | `00-首页/工作日志/YYYY-MM-DD.md` |

### 2.1 使用原则

1. 先用 `type` 做一级分层，再用 `role` 做二级分流。
2. 如果一页本质上是“给人读规则/导航/框架/历史分析”的，就归到 `系统`，不要再细分到 `说明`、`评估`、`方法论` 之类的 `type`。
3. `工作日志.md` 总索引页属于系统层，因此保持 `type: 系统`；只有每天的日志正文才用 `type: 工作日志`。
4. 任务卡体系单独维持 `type: 活跃任务卡`，不要与系统页或日志页混用。

## 四、不建议继续新增的近义词

以下叫法目前不建议再新增或恢复：

- `Agent最小协议`
- `Agent速查`
- `工具页`
- `说明页`
- `历史记录`
- `入口说明`
- `索引`
- `方法论`
- `评估`
- `课件索引`
- `备课框架`
- `命题分析`
- `日志`

原因：这些词要么过宽，要么与现有词表重叠，后续不利于 Dataview / Bases / 巡检稳定筛选。
