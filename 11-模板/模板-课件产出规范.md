# 课件产出规范

> 定位：从 Markdown 知识库生成 Word/PPT 教学材料的技术规范与样式标准。依据 2026-06-08 与 2026-06-16 两轮试点迭代及后续 v2 管线实践总结。
>
> 适用范围：`04-课件/试点产出/`、`06-学生侧材料/讲义/`（含 `试印版/`）中的所有 `.docx / .pptx` 文件。
>
> **当前生效管线**：`11-模板/scripts/build-all-handout-docx.py`（三段式：Preprocess → Pandoc → Postprocess）

---

## 一、总体原则

1. 先读备课框架、知识边界、教材提炼、网课提炼，再做正式产出。
2. 教师稿、学生讲义、PPT 各司其职，不互相替代。
3. PPT 服务投影与课堂节奏，不是“把讲义全文搬上屏幕”。
4. 公式、化学式、上下标、图片必须经过可视化检查，不能只看源码觉得“差不多”。
5. 图片只补有教学价值的内容，不用装饰图凑页。
6. 内容充实优先靠教材主干、网课高价值讲法、KP / 专题复用、题库落题，不靠空泛扩写凑字数。

## 二、字体规范

### 2.1 默认字体

| 场景 | 字体 | 说明 |
|:---|:---|:---|
| 中文正文 | **宋体 (SimSun)** | Word / 讲义默认口径；PPT 中文见 2.3 |
| 中文标题 | **黑体 (SimHei)** | Word / 讲义默认口径；PPT 中文见 2.3 |
| 英文 / 数字 / 变量 | **Times New Roman** | 保持公式与变量风格 |

### 2.2 python-docx 字体配置参考

```python
# 字体常量（docx_utils.py）
CN_BODY_FONT = "SimSun"       # 宋体 — 中文正文 / 表格正文
CN_HEAD_FONT = "SimHei"       # 黑体 — 中文标题 / 表头
EN_FONT = "Times New Roman"    # 英文 / 数字 / 变量

# 设置 run 字体（关键：同时指定 ascii + eastAsia）
def set_run_font(run, cn_font=CN_BODY_FONT):
    run.font.name = EN_FONT                        # ascii / hAnsi
    rpr = run._element.rPr
    rpr.rFonts.set(qn("w:eastAsia"), cn_font)     # 中文字体
    rpr.rFonts.set(qn("w:ascii"), EN_FONT)
    rpr.rFonts.set(qn("w:hAnsi"), EN_FONT)
    # 删除主题引用（防止 Word 回退为等线）
    for attr in ("w:eastAsiaTheme", "w:asciiTheme", "w:hAnsiTheme"):
        key = qn(attr)
        if key in rpr.rFonts.attrib:
            del rpr.rFonts.attrib[key]
```

每个 `TextRun` / `run` 都要显式设置字体，不依赖默认值。

### 2.3 PPT 字体

```css
font-family: 'SimHei', 'SimSun', 'Times New Roman', serif;
```

> 口径：`PPT` 中文不再默认使用 `Microsoft YaHei`。从现在起，中文部分改为 **黑体 / 宋体体系**；公式中的西文字母、变量、单位继续保持 `Times New Roman` 风格。

### 2.3.1 PPT 字体硬约束

- `PPT` 不能只写一个混合 `font-family` 然后交给系统自己决定。
- 所有可编辑文本对象默认都要按“**中文标题 = 黑体，中文正文 = 宋体，英文 / 数字 / 变量 / 分子式 / 西文标签 = Times New Roman**”去显式处理。
- 若某一页因投影距离、字号或信息密度原因，中文正文改用黑体，允许，但必须在加工页或脚本注释中说明原因。
- 教师提示框、页序建议、底部提醒条、窄栏侧注、卡片内两行以上小字，默认视为“投屏敏感区”。
- 投屏敏感区若出现宋体发虚、支撑力不足、窄栏换行松散，修正顺序为：**先缩句，再加字号，必要时切黑体**。
- 尤其以下内容，默认按 `Times New Roman` 处理，不得继续偷懒混在纯中文字体里：
  - 英文标题角标，如 `LESSON MAP`、`COMPARE 1`
  - 分子式 / 离子式中的西文字母，如 `CO₂`、`H₂O`、`BF₃`
  - 变量与符号，如 `x`、`μ`、`Δχ`
  - 英文缩写、单位中的西文部分
- 若当前工具链难以在同一文本框内完美中英混排，优先做法是：
  1. 拆成多个文本 run
  2. 必要时拆成多个文本框
  3. 最后才考虑图片兜底
- **禁止**出现“中文口径已经对了，但英文和分子式仍整块沿用中文字体”的交付物。
- **禁止**继续默认把整套 `PPT` 中文全部做成 `Microsoft YaHei`。

### 2.4 禁止事项

- ❌ Arial
- ❌ Calibri
- ❌ 未显式设置字体，直接吃系统默认

## 三、公式与化学式规范

### 3.1 默认策略

1. **Word 正式讲义 / 教师稿（通过 build-all-handout-docx.py 管线）**
   - Markdown 源文件中的 `$...$` / `$$...$$` LaTeX 公式由 Pandoc 自动转换为 OMML（Word 原生数学对象），完全可编辑
   - 短化学式、离子式、上下标优先用源码中的 LaTeX 表达（如 `$\ce{CO2}$`，预处理时转为 `$\text{CO2}$`）
   - 只有复杂排版明显失真时，才退回公式图片
   - 管线在 Preprocess 阶段自动将 `\ce{}` 转换为 `\text{}` 以确保 OMML 兼容

2. **PPT**
   - 能用文本框和标准上下标实现的，优先不用图片
   - 复杂公式、曲线图、难稳定排版的表达式，可用图片或矢量资源
   - 所有公式都要肉眼检查字符间距和上下标位置

### 3.1.1 PPT 公式专项口径

- `PPT` 里的公式不只检查“能不能显示”，还要检查“像不像教材和 Word 正文里的标准公式”。
- 重点检查四类问题：
  - 字母和数字之间空得过宽
  - 运算符与相邻字符脱节
  - 上标 / 下标离主体过远
  - 化学式数字没有正确落成下标、电荷没有正确落成上标
- 对以下表达式默认提高一档验收标准：
  - `E_k`、`K_b`、`K_f`、`ΔT_b`、`ΔT_f`
  - `CO₂`、`CH₂Cl₂`、`XeF₄`、`Fe²⁺`
  - 分数、根号、括号外指数、离子电荷
- 若文本框原生排版仍明显不自然，应改用：
  1. 更细的 run 拆分
  2. 预渲染公式图
  3. 重新布局，避免在过窄区域里硬塞公式

### 3.2 正式成品的硬约束

- 禁止保留 `_` 式伪下标，如 `K_b`、`ΔT_f`
- `E_k` 等量符下标应显示为真正下标
- 化学式数字默认一律做下标，如 `CO₂`、`H₂SO₄`
- 电荷数、分数幂、根号、括号外指数必须做成真正上标/下标
- 若使用图片公式，必须检查：
  - 数字与字母之间是否留白过大
  - 符号与相邻字符是否松散
  - 上标、下标是否离主体过远
  - 化学式下标数字是否像正常排版，而不是漂浮

### 3.3 图片公式渲染管线（备选方案）

当 OMML 无法满足排版精度时，使用以下管线生成公式图片：

```text
LaTeX 源码 -> MathJax-full (TeX->SVG) -> Sharp (SVG->PNG, 200 DPI) -> docx-js ImageRun / PPT asset
```

### 3.4 何时使用图片公式

| 场景 | 方式 | 理由 |
|:---|:---|:---|
| 展示公式（独立成行） | 优先 OMML，必要时图片 | 先保留可编辑性 |
| 行内提及 | 文本 | 阅读流更自然 |
| 短化学式 | 文本 | 直观、轻量 |
| 复杂化学方程式 | OMML 或图片 | 以稳定显示为准 |
| 表格单元格内 | 文本优先 | 图片难对齐 |

### 3.5 公式渲染脚本（备选）

> ⚠️ 当前主讲义管线（build-all-handout-docx.py）已通过 Pandoc 实现 OMML 原生公式，本节脚本仅用于需要图片兜底的备选场景。

**脚本位置**：`.claude/render-eqs.js`

```js
const container = mj.convert(latex, { display });
const svgNode = adaptor.firstChild(container);
let svg = adaptor.outerHTML(svgNode);
await sharp(Buffer.from(svg), { density: 200 }).png().toFile(fp);
```

### 3.6 docx-js ImageRun 用法

```js
new ImageRun({
  data,
  transformation: {
    width: 250,
    height: 50
  }
});
```

推荐宽度：

| 公式类型 | 推荐宽度 |
|:---|:---:|
| 简单（pV=nRT） | 250 px |
| 中等（M=ρRT/p） | 300 px |
| 复杂推导 | 350-400 px |

## 四、内容规范

### 4.1 三类产物分工

| 产物 | 核心职责 | 不应承担的职责 |
|:---|:---|:---|
| 教师上课大纲 | 课时分配、边界控制、板书/PPT 分工、教学提醒 | 不把学生正文整段复制进去 |
| 学生讲义 | 完整学习材料、推导、例题、易错点、作业 | 不承担教师侧边界提示 |
| PPT | 投影呈现、节奏推进、图表/题干展示 | 不做全文讲义上屏 |

### 4.2 学生讲义必须包含

| 内容 | 说明 |
|:---|:---|
| 学习目标 | 用“学完本节后你能……”句式 |
| 知识内容（含必要推导） | 不能只抛结论 |
| 课堂例题（含解答） | 建议 3-5 道，标注难度 |
| 核心结论 | 讲义末尾归纳 |
| 易错点预警 | 表格化表达 |
| 课后作业 | 必做 + 选做 |
| 学生自查清单 | 便于课后复盘 |
| 教材关联 | 对应章节或板块 |

### 4.2.1 学生讲义“五件套”底线

每个核心板块默认检查以下五项，缺一项就要说明是否故意留到板书或后续课时：

1. 定义
2. 原理或推导
3. 例题
4. 易错点
5. 练习

> 目标不是机械凑栏目，而是防止出现“只有概念没有题”“只有题没有原理”“只有总结没有训练”的空壳讲义。

### 4.3 建议纳入的拓展内容

在边界允许时，可纳入书上和网课中稳定覆盖、且不明显超纲的内容，如：

- Graham 扩散定律
- 气体分子运动论
- 温度的统计意义
- 速率 / 能量分布的定性图景
- van der Waals 方程的基础计算
- 依数性的完整表达

### 4.3.1 来源使用优先级

1. 教材与备课框架：确定主干内容和顺序
2. KP / 专题：复用已有结构化资产
3. 网课资料：补充讲法、类比、易错提醒
4. 题库 / 真题链：提供例题、变式、练习
5. 图片资产：提供图景和投影支持

> 考纲主要用来防失控，不用来机械删掉教材里本就值得讲的内容。

### 4.4 学生讲义禁止包含

- ❌ “本节深度边界”
- ❌ “本节暂不要求掌握”
- ❌ 仅有标题、没有实际内容的空壳板块

### 4.5 教师上课大纲必须包含

| 内容 | 说明 |
|:---|:---|
| 课时分配表 | 精确到分钟更佳 |
| 课前准备清单 | 学生前置知识 + 教师准备项 |
| 教学重难点表 | 重点/难点 + 说明 |
| 教学过程 | 含切入点、预警、例题链、边界决策 |
| 板书框架 | 课堂结构骨架 |
| 下节衔接 | 下一课接口 |

### 4.6 PPT 内容密度规范

- 一页一焦点：定义页、图景页、公式页、例题页、小结页尽量分开
- 优先改写成短句、表格、步骤、图
- 一页同时出现大段文字、长推导、例题全文，默认判定为过载
- 图片只补理解明显增益的内容，如教材分布图、实验图、结构图、相图、能量分布图

### 4.7 中文引号处理

在 docx-js 脚本中统一使用 `「」`，避免智能引号导致 JS 语法错误。

## 七、渲染与验收

### 7.1 讲义生成管线口径

- **默认管线**：`11-模板/scripts/build-all-handout-docx.py`
  - 三段式架构：Preprocess（YAML/wiki-link 预处理）→ Pandoc（Markdown→docx，LaTeX→OMML）→ Postprocess（python-docx 字体/样式修正）
  - 资源解析：直接从 vault 根目录 / `media/` 取图，不再为每份讲义复制整库图片
  - 字体口径：中文正文=宋体 SimSun、中文标题=黑体 SimHei、英文/数字/变量=Times New Roman
  - 公式策略：优先 OMML（Word 原生数学对象），备选图片公式
  - 质检闭环：可加 `--render-preview`，自动调用 `render_docx_windows.py` 输出逐页 PNG / PDF
- 运行前确保 Word 未打开目标文件（Windows 文件锁），可通过复制 `.tmp.docx` 后重命名绕过

### 7.2 最低验收动作

- 至少完成 1 轮 render 或导出后的可视化检查
- 逐项检查：
  - **公式**：OMML 公式可编辑（双击检查）、间距正常、上下标位置正确
  - **化学式**：`\ce{}` 预处理为 `\text{}` 后字符是否完整、下标数字是否落位
  - **表格**：是否溢出、单元格内换行是否合理
  - **字体抽样**：中文正文→宋体、中文标题→黑体、英文/变量→Times New Roman（选中 text 查看字体属性）
  - 图片是否清晰、是否真的有教学价值
- 若有 `PPT`，额外强制检查：
  - 英文标题角标是否已切到 `Times New Roman`
  - 中文标题是否已切到黑体、中文正文是否已按宋体 / 黑体口径落实
  - 教师提示框、页序建议、侧边提醒等小字号区是否单独做了投屏检查
  - 分子式 / 离子式 / 变量是否仍误用纯中文字体
  - 关键页中公式是否仍存在”空得过宽、上下标飘远”的问题
- 有问题时必须回写到加工页或 Markdown 源文件，而不是只在对话里口头说明
