# MD 到产品 PDF 落地方案

## 一句话目标

把 `04-课件/学生讲义/` 下的讲义，从“能编译”推进到“可稳定量产、可持续维护、可交给 Agent 使用”的产品级 PDF 管线。

## 一、当前判断

现有体系已经具备可用雏形：

- 有代表性样板：[[原子结构-超级充实版（自学完整）]]
- 有主编译脚本：`11-模板/scripts/convert_handout_to_pdf.py`
- 有导言模板：`11-模板/scripts/chemistry-preamble.tex`
- 有图片 Lua filter：`11-模板/scripts/wrap_images.lua`
- 有操作文档与策略文档

当前主要问题不是“完全不能出 PDF”，而是 **规范层、实现层、文档层还没有完全对齐**：

1. **规范与样板有漂移**：目标规范要求 `*图 N ...*`，现有样板仍有 `<center>图 ...</center>`。
2. **文档与实现漂移已收口**：图片缓存已统一为 workspace-local `11-模板/scripts/.chem_media/`，主手册与脚本实现现已对齐。
3. **任务边界不够清晰**：编译任务、规范化任务、模板开发任务混在一起，容易让 Agent 做出错误动作。

## 二、核心决策

### 1. 把体系拆成五层

| 层级 | 负责什么 | 当前载体 |
|:---|:---|:---|
| 内容层 | 讲义到底写什么 | `04-课件/学生讲义/*.md` |
| 规范层 | 讲义允许怎么写 | [[11-模板/MD讲义格式参考（原子结构样板）]] |
| 转换层 | MD 如何转成稳定 LaTeX | `convert_handout_to_pdf.py` + `wrap_images.lua` |
| 模板层 | PDF 长什么样 | `chemistry-preamble.tex` |
| 运行层 | Agent 如何安全使用 | 使用手册 + 一键说明 |

### 2. 把“目标规范”和“兼容层”分开

- **目标规范**：新稿、重构稿必须遵守的写法。
- **兼容层**：为了让历史讲义先能出 PDF，脚本暂时兜底的旧写法。

这一步非常关键。否则会出现“文档说不要用 `<center>`，样板却还在用”的长期混乱。

### 3. 把“编译”和“修源”拆成两类任务

- **编译任务**：默认不自动改源文件，只负责出 PDF、报错、验证。
- **规范化任务**：人工明确发起，允许修改讲义 Markdown / 图片命名 / 图注写法，使之逐步收敛到目标规范。

## 三、建议采用的最终工作流

### A. 新讲义 / 重构讲义

1. 先按 [[11-模板/MD讲义格式参考（原子结构样板）]] 编写。
2. 图片统一放在 vault 根目录 `media/`。
3. 图注统一写成 `*图 N 描述*`。
4. 公式统一走 `$...$` / `$$...$$` / `\ce{...}`。
5. 完成后再进入编译流程。

### B. 旧讲义快速出 PDF

1. 保留旧稿现状。
2. 运行 `convert_handout_to_pdf.py`。
3. 利用兼容层处理 `<center>`、旧图片名、别名映射等历史问题。
4. 出图后再决定是否发起“规范化任务”。

### C. 日常 Agent 执行

1. 先读 [[11-模板/PDF生成策略-Agent一键说明]]。
2. 再读 [[11-模板/LaTeX PDF管线使用手册]]。
3. 只在用户明确要求“修规范/改源文件”时，才进入内容修订。
4. 否则默认只编译、只验证、只报告问题。

## 四、落地阶段

### Phase 1：冻结输入规范

目标：以后新讲义不再扩散 legacy 写法。

完成标准：

- `MD讲义格式参考` 明确区分“目标规范”与“当前兼容”
- 新增讲义默认不用 `<center>`
- 新增讲义默认不在图注中写来源尾注
- 新增讲义默认使用稳定英文图片名

### Phase 2：稳定编译器

目标：把脚本从“手工调通”变成“稳定工具”。

完成标准：

- `convert_handout_to_pdf.py` 的用户可见入口固定
- 编译日志、缺图、页数、图片数输出稳定
- 明确哪些 legacy 写法继续兼容，哪些直接报错
- 补一个轻量 lint 清单，编译前先检查明显违规项

建议优先检查：

- `![[media/...]]` 是否存在
- `$...$` / `$$...$$` 是否闭合
- 表格是否误放在 blockquote 内
- 图注是否跟在图片后
- 是否残留高风险 HTML

### Phase 3：稳定产品模板

目标：把 PDF 的视觉和版式收敛成“可交付产品”。

完成标准：

- 章节标题、目录、页眉页脚样式固定
- 图片宽度规则固定
- 表格拥挤场景有明确策略
- callout 在 PDF 中有一致表现
- 选定 2 到 3 本样板讲义作为回归集

建议回归集：

- 原子结构：公式 + 图 + 趋势图最全
- 分子结构基础：结构图、流程图多
- 配位化合物基础：图表与概念辨析密度高

### Phase 4：收口运行文档

目标：让 Agent 和人都知道“什么时候怎么用”。

完成标准：

- 使用手册只讲“当前真实实现”
- 一键说明只讲“最短安全路径”
- 长策略文档只保留原理、坑点和设计决策
- 三份文档之间不再互相打架

## 六、我对现有方向的结论

方向本身是对的，而且已经过了“验证可行性”的阶段，下一步应该进入“产品化收口”阶段。

最值得坚持的三件事：

1. **先规范 Markdown，再追求一键生成。**
2. **把兼容层留在脚本里，不让 legacy 写法继续污染新稿。**
3. **把文档写成“真实实现说明”，不要写成理想状态说明。**

## 七、近期执行建议

按优先级，我建议这样推进：

1. 先按本文档和 `MD讲义格式参考`，把“目标规范 vs 当前兼容”彻底说清。
2. 再把 `LaTeX PDF管线使用手册` 与当前脚本实现对齐。
3. 之后补一个编译前 lint 说明或脚本。
4. 最后才考虑统一入口、批量化和更漂亮的视觉样式。

如果后面继续推进，我建议把下一项工作定义成：

**“做一个讲义 PDF 预检器（lint）”**，因为它能最快把“规范”真正落到流水线上。
