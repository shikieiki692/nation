# 模板：合并迁移计划

> 📋 **使用方式**：Agent 8 在 [[模板-差异对照表]] 通过用户审核后，按本模板生成具体执行计划。
> 落地路径：`08-待审核区/<日期>-<操作>-迁移计划.md`
>
> 🎯 **目的**：把合并/弃用拆成可逐步执行、可逐步回滚的小步骤，每一步都明确「执行什么 / 检查什么 / 失败如何回滚」。

## 一、计划概览

### 1.1 目标
<!-- 1-3 行说明完成本计划后达成什么状态 -->

例：
> 把 [[Lawesson 试剂]]（带空格变体）合并到 [[Lawesson试剂]]（标准名）。
> 完成后：① 12 处反向引用全部指向 target；② source 移到弃用归档；③ target 包含 source 的全部独有内容。

### 1.2 范围统计
- 涉及 KP：source <N> 个 + target 1 个 = <N+1> 个
- 反向引用待修复：<M> 处，跨 <K> 个文件
- 预计步骤数：<S> 步

### 1.3 风险等级
- 🟢 低：仅 1 个 source，反向引用 < 5 处，content 完全可吸收
- 🟡 中：2-3 个 source，反向引用 5-20 处，部分 content 取舍
- 🔴 高：> 3 个 source，反向引用 > 20 处，跨多模块影响

## 二、执行步骤（按顺序）

> 每一步都按「检查前置 → 执行 → 检查后置 → 提交日志」四步走。
> 如任何一步后置检查失败 → **立即停止**，不执行后续，按第七节回滚。

### Step 1：备份 source 全文
**目的**：防止任何阶段意外丢失内容，便于回滚。

**前置检查**：
- [ ] source KP 文件存在
- [ ] 已读 source 全文

**执行**：
1. 把 `<source 路径>` 复制到 `08-待审核区/弃用归档/<source 文件名>.bak`
2. 用 PowerShell：`Copy-Item "<source>" "08-待审核区/弃用归档/<source>.bak"`

**后置检查**：
- [ ] .bak 文件存在
- [ ] 字数与原文一致（用 `Measure-Object -Line` 或 `Read` 比对）

**日志**：在 [[工作日志]] Phase 2 记录「✅ 已备份 <source>」

---

### Step 2：合并独有信息到 target

> 仅合并模式需要；弃用模式跳过此步。

**前置检查**：
- [ ] 已读 target 全文
- [ ] 已读差异对照表第 4.1 节（source 独有信息清单）
- [ ] 已确认每条独有信息要插入 target 的哪一段

**执行**（每条独有信息一次 Edit）：

| 内容点 | 来自 source 段落 | 插入 target 段落 | 操作 |
|---|---|---|---|
| <内容 1> | ③ 关键性质 | ③ 关键性质 | Edit, 段末追加 |
| <内容 2> | ④ 机理 | ④ 机理 | Edit, 替换 target 单薄段 |
| <内容 3> | ⑤ 例题 | ⑤ 例题 | Edit, 段末追加 |
| ... | ... | ... | ... |

**后置检查**：
- [ ] target 字数从 ~<旧> 增加到 ~<新>，符合预期
- [ ] target 15 段结构完整未破坏
- [ ] target 的 `version` 符合当前模板；若仍保留 `template_version`，其值与 `version` 一致
- [ ] target 化学式/wikilink 渲染正常（[[经验沉淀]] 1/2/3 类规则）
- [ ] target frontmatter 的 `updated` 已改为今日

**日志**：在 [[工作日志]] Phase 2 记录「✅ 已合并独有信息到 <target>」

---

### Step 3：合并 aliases

**前置检查**：
- [ ] 已读差异对照表第 6 节（aliases 合并清单）

**执行**：
1. Edit target 的 frontmatter，把 `aliases` 字段改为合并去重后的列表
2. 注意：YAML 半角冒号（[[经验沉淀]] 1-1）

**后置检查**：
- [ ] aliases 字段格式合法（`Grep "^aliases:" -A 5 <target>`）
- [ ] 包含 source 的原始名（重要！这样老 wikilink 还能解析到）

**日志**：「✅ aliases 已合并：[A, B, C, D]」

---

### Step 4：批量更新反向引用

**前置检查**：
- [ ] 已读差异对照表第 5 节（反向引用清单），共 <M> 处
- [ ] 已记录每处的「文件 + 行号 + 上下文」

**执行**：

> ⚠️ 重要：必须**逐文件**改完所有反向引用，不能边改 source 边改引用（会出脏数据 [[经验沉淀]] 6-1）。

1. 对引用清单中的每一文件：
   - Read 该文件
   - 用 Edit 把 `[[<source 名>]]` 改为 `[[<target 名>]]`
   - 注意：如果同一行有 `[[source]]` 出现多次，分多次 Edit 或用 replace_all（慎用）
2. 改完一文件就标记进度：`✅ 已改 <文件名>（X→Y 处）`

**后置检查**（关键）：
- [ ] 跑全库 Grep 验证：`Grep -r "\[\[<source 全名>\]\]" --include="*.md"` 应为 0 命中
- [ ] 也要 Grep source 的旧 aliases：每个 alias 都要确认无残留
- [ ] 抽查 3 个文件，确认改后内容上下文通顺（没改错地方）

**如果发现遗漏**：
- 立即停止后续步骤
- 修复遗漏后重新跑后置检查
- 全部清零才能进 Step 5

**日志**：「✅ N 处反向引用已修复，跨 K 个文件」

---

### Step 5：移动 source 到弃用归档

**前置检查**：
- [ ] Step 4 后置检查全部通过（反向引用 0 残留）
- [ ] .bak 备份仍在

**执行**：
1. 用 PowerShell：
   ```powershell
   Move-Item "<source 原路径>" "08-待审核区/弃用归档/<source 文件名>"
   ```
2. Edit 移动后的 source，在文件顶端（frontmatter 之后）插入横幅：
   ```markdown
   > ⚠️ **本笔记已被合并到 [[<target>]]**，将于 <hard_delete_at> 硬删除。
   > 如有异议，请把本文件移回 `<source 原路径>` 并修复反向引用。
   ```
3. Edit source 的 frontmatter，添加：
   ```yaml
   deprecated: true
   replaced_by: <target 名>
   deprecated_at: <今日>
   hard_delete_at: <今日 + 7>
   ```

**后置检查**：
- [ ] 原路径已无 source 文件
- [ ] 弃用归档中存在 source 文件
- [ ] frontmatter 含 deprecated: true
- [ ] 横幅已添加

**日志**：「✅ source 已软弃用，归档至 08-待审核区/弃用归档/，hard_delete_at=<日期>」

---

### Step 6：更新索引与状态

**前置检查**：
- [ ] Step 5 完成

**执行**：
1. **[[知识点总索引]]**（如手工维护）：
   - Read 索引文件
   - Edit 删除 source 行
   - 如有「合并历史」段落，追加一条「<日期> source → target」
2. **[[状态摘要]]**：
   - Dataview 库存表自动更新（无需手工）
   - 如有「最近 3 次任务」段，由 Phase 4 归档时同步
3. **[[工作日志]]**：在 Phase 2 写完整执行流水

**后置检查**：
- [ ] 索引中无 source 残留
- [ ] [[状态摘要]] 库存表数字符合预期（subject 列对应数 -1）
- [ ] [[工作日志]] 当日条目完整

**日志**：「✅ 索引与状态已同步」

---

### Step 7：通知用户 + 进入观察期

**执行**：
1. 向用户报告：
   ```
   ## 合并完成报告
   - 主 KP：[[<target>]]
   - 已弃用：[[<source>]]（位于 08-待审核区/弃用归档/）
   - 反向引用修复：N 处
   - 硬删除日期：<hard_delete_at>
   
   📌 7 天观察期：如发现问题，把 source 移回原位置并通知 Agent 修复反向引用。
   ```
2. **不**自动启硬删除——必须等 7 天 + 用户/Agent 9 触发。

---

### Step 8（7 天后）：硬删除

**前置检查**：
- [ ] 当前日期 ≥ hard_delete_at
- [ ] 用户已确认无回滚需求（或 Agent 9 健康检查触发）
- [ ] .bak 备份仍在弃用归档

**执行**：
1. 用 PowerShell：
   ```powershell
   Remove-Item "08-待审核区/弃用归档/<source 文件名>" -Confirm:$false
   ```
2. .bak 备份保留 30 天后再删（多一道保险）

**日志**：「🗑️ <source> 已硬删除（合并到 <target>）」

## 四、回滚预案

### 4.1 哪些步骤可回滚
| 步骤 | 可回滚？ | 回滚方法 |
|---|---|---|
| Step 1 | ✓ | 删除 .bak |
| Step 2 | ✓ | Edit 撤回 target 改动 |
| Step 3 | ✓ | Edit 改回 target.aliases |
| Step 4 | ✓（按文件） | 逐文件改回 `[[source]]` |
| Step 5 | ✓ | 把 source 从归档移回原路径 + 改 deprecated 字段回 false |
| Step 6 | ✓（手工） | 索引/状态摘要手工修复 |
| Step 8 | ❌（永久） | .bak 也删后无法回滚——必须确认 |

### 4.2 全计划回滚流程
若执行到任意 Step 后发现问题需全回滚：
1. **从最近的 Step 开始反向操作**（Step 5 → Step 4 → ...）
2. 用 .bak 文件验证 source 内容完整
3. 全部回滚后，在 [[工作日志]] 记录「⚠️ 计划已回滚，原因：<原因>」
4. 把本计划文件 status 改为「已回滚」

### 4.3 部分回滚
如只是 Step 4 中 1-2 个文件改错：
1. 单文件 Edit 改回去
2. 不影响整体计划继续

## 模板使用注意事项

### 强制要求
- 计划必须**先用户确认再执行**（[[Agent提示词-知识库维护]] Agent 8 强约束）
- 每一步后置检查必须实际跑命令验证，不能"目测"
- Step 4（反向引用）必须 0 残留才能进 Step 5

### 简化使用
- 弃用模式（无 target）：跳过 Step 2、Step 3，第二节的 aliases 段也跳过
- 单 source 简单合并：可省略「执行进度跟踪」段，直接顺序执行

### 与 [[模板-差异对照表]] 配套
- 差异对照表 = "为什么要做、做后什么样"
- 迁移计划 = "怎么一步步做"
- 两者是同一次操作的两个文档，filename 共享日期与 source/target
